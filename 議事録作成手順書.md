# 議事録作成手順書（手動運用版）

Teams録画から議事録を作成し、Backlog全社共通（ALL）のドキュメントへ登録するまでの**手動操作手順**。実機で検証・確定した内容を反映している（現時点版）。まずは手動で正常に運用できることを目的とし、自動化は本手順が定着した後の拡張とする。

- 環境構築（FFmpeg・whisper.cpp・モデル・作業場所）：`環境構築手順書.md`
- 書式の詳細：`Markdownフォーマット定義.md`
- 実施結果・知見：`手順動作確認.md`
- 全体の要件：`議事録作成・公開運用_要件定義書.md`
- （将来）自動登録の設計：`Backlog通信定義.md`

作業の基点は **`C:\minutes`**（ASCII固定パス）。作業データは `C:\minutes\work\` に置く。

---

## 0. 事前準備

`環境構築手順書.md` に従い、以下が用意済みであること。

- FFmpeg（`ffmpeg -version` が通る）
- whisper.cpp 本体（`C:\minutes\tools\whisper\Release\whisper-cli.exe`）とモデル（`…\models\ggml-medium.bin`）
- 作業フォルダ `C:\minutes\work`
- 使用AI：手動貼り付け。外部AI・外部サーバー（さくらサーバー／さくらAIエンジン等）を使う場合は、事前に上長の確認・許可を得る
- Backlog 全社共通（ALL）ドキュメントへの登録権限（手作業は画面から。スクリプト自動登録＝方法A を使う場合は、各自のBacklog APIキーを環境変数 `BACKLOG_API_KEY` に設定する。**APIキーの発行・スペースのAPI有効化は上長の承認が前提**）

## 1. 録画を取得する（サイボウズ）

**保存場所**：会議の録画（.mp4）は**サイボウズ（cybozu.com）のファイル管理**に置かれている。
- 例：`https://tcs-s.cybozu.com/o/ag.cgi?page=FileIndex&...`

**取得方法**
1. サイボウズの該当フォルダを開き、対象の録画（.mp4）を**PCにダウンロード**する。
2. ダウンロードした `.mp4` を **`C:\minutes\work\`** に置く。**社内情報のため、外部同期される場所（個人のクラウド共有等）には置かない**。
3. この録画の**サイボウズURL**は、議事録の「録画リンク」にも使う（Step 4 の `meeting.txt` の `Link=` に記入）。フォルダに動画が複数あるときは、できれば動画1本を開いたときの個別URLを使う。

> 補足：録画はTeamsで作成後、サイボウズにアップロードして共有する運用。ダウンロードできない場合は、閲覧・取得の権限を管理者／開催者に確認する。

**音声トラックの確認**（ファイル名は実物に置換）
```powershell
$base = "C:\minutes"
$rec = "$base\work\（録画ファイル名）.mp4"
ffprobe "$rec"
```
→ `Duration: 00:xx:xx`（録画時間）と `Stream ... Audio:`（音声トラック）があることを確認。音声が無ければ処理を止め、録画の取得元を確認する。

## 2. 音声を抽出する（FFmpeg）

```powershell
$base = "C:\minutes"
$rec = "$base\work\（録画ファイル名）.mp4"
ffmpeg -y -i "$rec" -ar 16000 -ac 1 -c:a pcm_s16le "$base\work\transcript.wav"
```
- 出力 `transcript.wav` は16kHz・モノラル・16bit PCM。
- 入力ファイル名に日本語や記号が含まれても可（出力はASCII名の `transcript.wav`）。
- `ffmpeg` が「認識されない」と出たら、PATHを取り込み直す：
  `$env:Path = [System.Environment]::GetEnvironmentVariable("Path","Machine") + ";" + [System.Environment]::GetEnvironmentVariable("Path","User")`

---

## 3. 文字起こしする（whisper.cpp）

```powershell
$base = "C:\minutes"
& "$base\tools\whisper\Release\whisper-cli.exe" -m "$base\tools\whisper\models\ggml-medium.bin" -f "$base\work\transcript.wav" -l ja -otxt -osrt -ovtt -pp -of "$base\work\transcript"
Get-Content "$base\work\transcript.txt" -Encoding UTF8
```
- 出力：`transcript.txt`（AI議事録生成の元データ）、`transcript.srt` / `.vtt`（発言箇所の確認用）。
- `-l ja`（日本語）と `-pp`（進捗表示）を付ける。
- **所要時間の目安：録画時間の約1.5〜2倍（CPU実行）**。実行中は放置でよい。
- **txtはUTF-8**。読むときは `-Encoding UTF8` を付ける（付けないとPowerShellで文字化けする。ファイル自体は正しい）。

## 4. AIで議事録案を作成する

会議情報を入力するだけで、AIに貼るプロンプト（プロンプト本体＋文字起こし全文）を自動生成できます（方法A・推奨）。手作業でも組み立てられます（方法B）。

### 方法A（推奨）：スクリプトで自動生成

`transcribe.ps1` の文字起こし完了後、続けて `buildprompt.ps1` が自動で走る（**質問はされない＝非対話**）。会議情報は次のように決まる。

- **会議番号**：連番で自動採番（同じ録画をやり直しても増えない）
- **開催日**：録画ファイルの日時から自動
- **録画リンク・登壇者**：`C:\minutes\work\meeting.txt` に記入した値

初回は `meeting.txt`（会議番号・開催日は自動で入り、録画リンク・登壇者は空）が作られてnotepadで開く。**`Link=` に録画のサイボウズURL、`Speaker=` に登壇者名を書いて保存**し、もう一度 `buildprompt.ps1` を実行する。

```text
No=5
Date=2026/07/09
Link=https://tcs-s.cybozu.com/o/ag.cgi?page=FileIndex&...（録画のURL）
Speaker=渡辺 太郎
```

→ 完成プロンプトが `C:\minutes\work\ai_prompt.txt` に保存され、**クリップボードにコピー**＆**notepadで表示**される。会議名・登壇者を確認し、**承認済みAIに貼り付け**（Ctrl+V）して送信、出力されたMarkdownを保存する。

- 自動の会議番号・開催日がずれていたら、`meeting.txt` の `No=` / `Date=` に書けば**その値が優先**される。
- `buildprompt.ps1` は**単体でも実行可能**。`transcript.txt` があれば、**再文字起こしなしでプロンプトだけ作り直せる**。
  ```powershell
  powershell -ExecutionPolicy Bypass -File "C:\minutes\buildprompt.ps1"
  ```
- プロンプト本体は `prompt_template.txt`（UTF-8）が正本。会議名の固定部分（`システム設計×AI駆動開発勉強会`）や差し込み内容を変えるときはこのファイルを編集する。

### 方法B：手作業で組み立てる

1. 承認済みのAIを開く。**外部AI・外部サーバーを使う場合は、事前に上長の許可を得ていること。**
2. 下記プロンプトの「会議名」「開催日」「録画リンク」「登壇者」を実際の会議に合わせて書き換える（録画リンクは社内・閲覧のみの共有URL。無ければ「なし」）。
3. プロンプト末尾の `---` の間に、`transcript.txt` の中身（`Get-Content ... -Encoding UTF8` で表示したもの）を貼り付ける。
4. AIに送信し、出力されたMarkdownを保存する。

# この議事録の目的（前提）
- 目的は、会議に出ていない人や後から参照する人が、内容を正確に把握できるようにすることです。
- 逐語録ではなく要点を整理し、前提や背景も簡潔に補って、その場にいなくても理解できる粒度で書いてください。

# 会議情報（以下は私が指定します。会議概要のヘッダーはこの指定値を優先すること）
- 会議名: （例：第〇回システム設計×AI駆動開発勉強会）
- 開催日: （例：yyyy/mm/dd）
- 開催方法: Teams
- 録画リンク: （社内・閲覧のみの共有URL。無ければ「なし」と書く）
- 登壇者: （その会のスピーカーの正式名。複数可。無ければ空欄）

# 発言者名の扱い（重要）
- この文字起こしは話者分離をしていないため、話者名（例：「〇〇:」）は誤り・欠落・実在しない名前の可能性があります。
- ただし、後から誰の発言かを追いやすくするため、文字起こしに現れる発言者名は、誤りの可能性があってもそのまま本文に記載してよい。
- これらの名前は未確認である前提とする。正式な氏名はTeamsトランスクリプトで後から確認・修正する運用のため、AIは名前が正しいかの判断や修正はしなくてよい。
- 会議概要の「登壇者」やTODOの「担当」は、指定があればその値を優先して使う。指定が無ければ文字起こしから読み取れる名前を記載してよい。確定できない場合は名前の末尾に「（要確認）」を付す。
- ただし、機微な文脈（異動・評価・労務・懲戒など）の個人名は、本文に断定して書かない（後述の条件・6.11に従う）。

# 決定事項・TODO・確認事項の定義（切り分け）
- 決定事項：その会で確定した事実のみ。未確定・検討段階のものは入れない。
- TODO・対応事項：会議後に誰かが行う具体的なアクション。担当・期限が不明なら「要確認」。
- 確認事項・要確認事項：事実かどうかの裏取りや、公開可否の判断が必要なもの。

# 条件
- 会議に出ていない人が把握できる粒度に要点を整理する（逐語録にしない）
- 雑談・言い直し・関係のない発言は省略する（ただし決定に至った経緯など、理解に必要な背景は残す）
- 未確定のことを「決定事項」として書かない（迷う場合は「確認事項」に回す）
- 不明な箇所は推測で埋めず「要確認」と書く
- 動画は見直しておらず、文字起こしのみを根拠にしている前提で書く
- 文字起こし特有の誤り（会社名・数字・日付・金額・固有名詞・社内用語・同音異義語）が疑われる箇所は、本文で断定せず「要確認」または確認事項に回す（発言者の人名は上記「発言者名の扱い」に従い、誤りの可能性があっても記載する）
- 確認事項は重複を整理し、公開可否に影響する重要なものを優先する（同種の指摘を無闇に増やさない）
- 個人情報・機密・社外秘に見える内容は本文に断定して書かず、確認事項に回す。具体例：機微な文脈の個人名（異動・評価・労務・懲戒など）／顧客・取引先名／未公開案件／契約・金額・単価／障害・インシデントの詳細／セキュリティ情報。迷ったら本文に載せず確認事項へ回す（ただし発言者・登壇者・TODO担当などの氏名は、未確認でも本文に記載してよい）
- 確認事項に回す場合も、具体的な氏名・会社名・案件名・金額はそのまま書かず、抽象化して記載する
- 文体は常体（〜である／〜した）で統一する

# Markdown記法のルール
- 見出しは #、##、### を使う
- 決定事項・TODO・確認事項は必ずMarkdownの表で書く
- 箇条書きは - を使う / 番号付きは 1. 2. を使う
- 強調（太字など）は必要最小限にする
- HTMLタグ・画像は使わない / 外部リンクは本当に必要なときだけ
- コードブロックはコマンドや設定値を書くときだけ使う
- 該当する内容がない章は、空欄にせず「該当なし」と書く（ただし会議概要の各項目は下記テーブルの指示に従う）
- 担当者・期限が文字起こしから判断できない場合は「要確認」と書く

## 7. Backlogに登録する

**公開前確認（Step 6）を通過した議事録のみ**を登録する。方法A（スクリプト・推奨）と方法B（手作業）がある。

### 方法A（推奨）：スクリプトで自動登録（`register.ps1`）

BacklogのドキュメントAPIで、Markdownをそのまま登録する。

**事前準備（初回のみ）**
- `setup.ps1` 実行済み（`register.ps1` と `backlog.config.txt` が `C:\minutes` に配置される）。
- 自分のBacklog APIキーを環境変数に設定（**発行・API有効化は上長承認が前提**）：
  ```powershell
  setx BACKLOG_API_KEY "＜自分のAPIキー＞"
  ```
  → 設定後は**新しいPowerShell**から有効。
- `C:\minutes\backlog.config.txt` に登録先（Space / ProjectId / ParentId）が入っていること（サンプルから配置済み。**APIキーはここに書かない**）。

**登録手順**
1. Step 5 で整えた議事録を **`C:\minutes\work\minutes.md`** として保存（別名なら実行時に `-Path "＜ファイル＞"` で指定）。
2. まず送信せず内容確認（ドライラン）：
   ```powershell
   powershell -ExecutionPolicy Bypass -File "C:\minutes\register.ps1" -DryRun
   ```
   → タイトル（Markdown先頭のH1）・Space・ProjectId・ParentId・`ApiKey: set` を確認。
3. 問題なければ本登録：
   ```powershell
   powershell -ExecutionPolicy Bypass -File "C:\minutes\register.ps1"
   ```
   → 送信前に確認を聞かれるので `y` で登録。成功すると `Document id` が表示される。
4. Backlogで、親ドキュメント配下に作成されていること・**日本語が化けていないこと**・表示崩れが無いことを確認する。

**注意**
- **タイトルはMarkdownの先頭H1**から自動取得する（命名規則 `YYYYMMDD_会議名_議事録` どおりのH1にしておく）。
- このアカウントは**追加はできるが削除はできない**（削除は管理者権限）。**誤登録すると自分では消せない**ため、必ず「ドライラン → 確認プロンプトで `y`」の順で慎重に行う。誤登録時は管理者にドキュメントIDを添えて削除を依頼する。
- 現状、登録前の自動重複チェックは未実装。実行前にBacklog画面で同名が無いか確認するのが安全。

### 方法B：手作業で登録する

1. Backlog を開き、**全社共通（ALL）→ ドキュメント → 登録先の親ドキュメント**を開く。
2. 同じタイトルの議事録が既に無いか確認する（重複登録を避ける）。
3. 「ドキュメントを追加」し、**タイトル**に命名規則どおりの名前を入力する（例：`20260703_第1回システム設計×AI駆動開発勉強会_議事録`）。
4. Step 5 で整えた**Markdown本文をそのまま貼り付ける**。
5. 保存し、**表示が崩れていないか**（見出し・表・箇条書き）を確認する。

## 8. うまくいかないときは（実機で判明した注意点）

| 状況 | 対応 |
| --- | --- |
| `ffmpeg` が認識されない | PATHを取り込み直す（`$env:Path = ...` の1行）。または新しいPowerShellを開く |
| whisper で日本語パスが文字化け／ファイルを開けない | 作業場所をASCIIパス（`C:\minutes`）にする。本手順はその前提 |
| `transcript.txt` が文字化けして見える | `Get-Content ... -Encoding UTF8` で読む（ファイル自体は正しい） |
| `curl` が別コマンドとして動く | PowerShellでは `curl.exe` を使う |
| 録画をチャットからダウンロードできない | 権限制限。開催者にダウンロード可で共有を依頼 |
| 音声トラックが無い／FFmpeg失敗 | 録画の取得元・パス・音声有無を確認（`音声テキスト化方式選定.md`） |
| whisper.cpp 実行エラー | 実行ファイル・モデル・WAV形式を確認（同上） |
| AI出力が不十分・崩れている | プロンプトを調整して再生成する |
| Backlog登録が失敗する | 少し時間をおいて再登録、または管理者に相談 |

## 9. この手順の位置づけ（将来の自動化との関係）

本手順は**手動で正常動作する運用**を確立するためのもの。動作が安定した後、以下を段階的に自動化していく想定（設計は既存の定義書に温存）。

- 音声抽出〜文字起こし〜Markdown整形の自動実行
- 公開前確認結果の `approval.json` 管理・本文ハッシュ照合
- Backlog APIによる自動登録：**登録自体は `register.ps1` で実装済み**（Step 7 方法A）。残りの重複確認・排他制御・登録後確認・承認ゲート連携は今後の拡張

自動化にあたり、録画・音声・文字起こし・議事録案を外部AI／外部サーバーで扱う場合は、**上長の確認・許可を前提**とする。

## 付録：スクリプトで効率化する（任意）

Step 0（環境構築）・Step 2〜3（音声抽出・文字起こし）・Step 4のプロンプト組み立ては、スクリプトで実行できます。**手順の理解は本文で、実行はスクリプトで**という位置づけです（スクリプトはリポジトリで管理し、レビュー済み）。

### 環境構築：`setup.ps1`
新しいPCで、FFmpeg・whisper.cpp・モデル・作業フォルダを一括で用意し、`transcribe.ps1`・`buildprompt.ps1`・`register.ps1`・`prompt_template.txt`・`field_labels.txt`・`backlog.config.sample.txt` を `C:\minutes` へ配置します（何度実行しても安全＝再実行で壊れない）。`backlog.config.txt` が無ければサンプルから自動生成します（既存は上書きしません）。
```powershell
powershell -ExecutionPolicy Bypass -File "（プロジェクトフォルダ）\setup.ps1"
```
※初回のFFmpeg導入後は、`ffmpeg` を使うために**新しいPowerShellを開く**と確実です。

> **注意：作業フォルダ名を変える／別PCの既存フォルダを移すときは、先に実フォルダをリネーム／配置してから `setup.ps1` を実行する。** 順番を逆にすると、`setup.ps1` が「フォルダが無い」と判断して**モデル等を一から再ダウンロード**し、既存のデータ（録画・文字起こし）は引き継がれません（例：`C:\gijiroku` → `C:\minutes` へ移す場合は、先に `Rename-Item "C:\gijiroku" -NewName "minutes"` を実行）。

### 音声抽出＋文字起こし：`transcribe.ps1`
録画（.mp4）を `C:\minutes\work` に置いて実行すると、音声抽出→文字起こし（Step 2〜3）を自動で行い、`transcript.txt` を表示します。
```powershell
powershell -ExecutionPolicy Bypass -File "C:\minutes\transcribe.ps1"
```
- work内に録画が**複数あるときは番号で選択**できます（1つなら自動）。
- **発言者名のヒント**：`C:\minutes\work\names.txt`（UTF-8）に参加者名を書いておくと、`--prompt` として渡され名前の誤変換が減ります（無くても動作）。日本語プロンプトはWindows環境により効きが変わるため、**最終的な名前はTeamsトランスクリプト等で要確認**。
- 失敗時（録画なし・ツール未導入・変換/文字起こしエラー）は、原因を表示して停止します。
- 文字起こし完了後、続けて `buildprompt.ps1` が自動で走り、`work\meeting.txt` の会議情報から**AIに貼るプロンプト**（`work\ai_prompt.txt`）を生成します（**非対話**。詳細は Step 4）。使用した録画の情報は `work\source.txt` に記録され、開催日の自動入力と連番採番に使われます。

### AIプロンプト生成：`buildprompt.ps1`
文字起こし後、`prompt_template.txt` に会議情報を差し込み、`transcript.txt` 全文を末尾に加えた**完成プロンプト**を `C:\minutes\work\ai_prompt.txt` に保存し、**クリップボードにコピー**＆**notepadで表示**します。**質問はされません（非対話）**。会議番号は連番で自動、開催日は録画の日時から自動、録画リンク・登壇者は `work\meeting.txt` に記入した値を使います（詳細は Step 4 方法A）。単体でも実行可能です。
```powershell
powershell -ExecutionPolicy Bypass -File "C:\minutes\buildprompt.ps1"
powershell -ExecutionPolicy Bypass -File "C:\minutes\buildprompt.ps1" -NoOpen   # notepadを開かない（自動化用）
```
- 会議名の固定部分や差し込み内容を変えるときは `prompt_template.txt`（UTF-8）を編集します。
- 表示ラベルは `field_labels.txt`（UTF-8）で定義しています（スクリプト本体は化け防止のためASCIIのみ、日本語ラベルは外部ファイルから読み込み）。
- 会議名・登壇者など日本語は notepad の表示で確認し、必要なら直してからAIへ貼り付けてください。

### まとめて実行：`run.ps1`
パイプラインの入口です。人手ゲート（AIで議事録作成＋公開前確認）の前後を1コマンドにまとめます。
```powershell
powershell -ExecutionPolicy Bypass -File "C:\minutes\run.ps1"                # 文字起こし→プロンプト生成まで
powershell -ExecutionPolicy Bypass -File "C:\minutes\run.ps1" -To register   # レビュー済み議事録をBacklog登録
```
- `-To prompt`（既定）は、文字起こし→AIプロンプト生成まで進み、人手ゲート（AIで議事録作成→`work\minutes.md` 保存→Step6確認）で停止します。そのあと `-To register` で登録します。
- 将来、承認済みAIのAPI利用が整えば、間にAI自動生成の段（`sendprompt.ps1`）を挟み、より一気通貫に近づけられる設計です。

### Backlog登録：`register.ps1`
公開前確認を終えた議事録（`C:\minutes\work\minutes.md`）を、BacklogのドキュメントAPIで登録します（詳細は Step 7 方法A）。登録先の設定は `backlog.config.txt`（Space / ProjectId / ParentId）、APIキーは各自の環境変数 `BACKLOG_API_KEY` で渡します。
```powershell
powershell -ExecutionPolicy Bypass -File "C:\minutes\register.ps1" -DryRun   # 送信せず確認
powershell -ExecutionPolicy Bypass -File "C:\minutes\register.ps1"           # 本登録（yで確定）
```
- 本登録は送信前に確認プロンプトが出ます（自動化・バッチ用に `-Force` で省略可）。
- APIキーは設定ファイルに書かず環境変数で渡します（configにキーがあると警告表示）。
- **チーム配布**：共有の登録先（Space / ProjectId / ParentId）はサンプルに含めて配布し、各自は自分のAPIキーを環境変数に設定するだけで使えます。Backlog上の作成者が本人として記録され、誰が登録したか追跡できます。
- このアカウントは追加はできても**削除は管理者権限**が必要です。誤登録防止のため、まず `-DryRun` で確認してください。

### スクリプトでカバーしない範囲
Step 4 のプロンプト組み立ては `buildprompt.ps1`、Step 7 のBacklog登録は `register.ps1` で自動化しました。一方、**AIへの送信・出力の保存（Step 4後半）〜整形・公開前確認（Step 5〜6）は手作業**です（AIは承認済みのものを使用、外部は上長承認）。特に**公開前確認（Step 6）は人が行う前提**で、`register.ps1` はその確認を終えた議事録を登録する位置づけです。`names.txt`（参加者名）はリポジトリに同梱し、`setup.ps1` 実行時に `C:\minutes\work` へ配置します（既にあれば上書きしません）。参加者が変わったら書き換えてください。
