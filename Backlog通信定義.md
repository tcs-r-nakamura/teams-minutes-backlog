## Backlogとの通信定義

### 1. 基本方針

本件では、作成した議事録をBacklogの全社共通（ALL）のドキュメント配下へ登録する。

登録方式は、Backlog画面からの手動登録ではなく、Backlog APIを使用した登録を基本とする。

ただし、全社員向けに公開する資料であるため、AIが生成した議事録を即時登録するのではなく、登録前に人間による公開可否確認を行う。

| 項目             | 方針                        |
| -------------- | ------------------------- |
| 登録先            | Backlog 全社共通（ALL）ドキュメント配下 |
| 登録方式           | Backlog APIによる登録          |
| 使用API          | Backlog Document API      |
| 登録対象           | 公開前確認済みのMarkdown議事録       |
| 即時自動公開         | 行わない                      |
| 登録前確認          | 必須                        |
| 重複登録           | 防止する                      |
| 同名ドキュメントの自動上書き | 行わない                      |
| 誤登録時の削除        | 管理者またはプロジェクト管理者が手動判断後に実施  |

---

### 2. 採用するAPI

本件で使用するBacklog APIは以下とする。

| No | 用途          | API                                     | 使用目的                                          |
| -- | ----------- | --------------------------------------- | --------------------------------------------- |
| 1  | 認証確認        | `GET /api/v2/users/myself`              | APIキーが有効か確認する                                 |
| 2  | プロジェクト確認    | `GET /api/v2/projects/{projectIdOrKey}` | `ALL` プロジェクトの実在、projectId、Document機能有効状態を確認する |
| 3  | ドキュメントツリー取得 | `GET /api/v2/documents/tree`            | 登録先の親ドキュメントIDを特定する                            |
| 4  | ドキュメント一覧取得  | `GET /api/v2/documents`                 | 同名議事録が存在しないか確認する                              |
| 5  | ドキュメント追加    | `POST /api/v2/documents`                | 確認済み議事録を登録する                                  |
| 6  | ドキュメント取得    | `GET /api/v2/documents/{documentId}`    | 登録後の確認に使用する                                   |
| 7  | ドキュメント削除    | `DELETE /api/v2/documents/{documentId}` | 誤登録時に管理者またはプロジェクト管理者が削除する                     |
| 8  | レート制限確認     | `GET /api/v2/rateLimit`                 | 補助APIとしてAPI制限状況を確認する（必須ではない）                   |

---

### 3. 認証方式

Backlog APIとの通信には、API Key方式を採用する。

本件は、特定のBacklogプロジェクト配下に議事録を登録する社内運用であるため、初期実装ではAPI Key方式を採用する。

OAuth 2.0は、複数ユーザーがそれぞれ認可して利用する正式な社内アプリケーション化を行う場合に検討する。

---

### 4. 設定値

Backlog連携で必要な設定値は以下とする。

| 設定名                          | 必須 | 内容                          |
| ---------------------------- | -- | --------------------------- |
| `BACKLOG_BASE_URL`           | 必須 | BacklogのベースURL              |
| `BACKLOG_API_KEY`            | 必須 | Backlog APIキー               |
| `BACKLOG_PROJECT_KEY`        | 必須 | 登録先プロジェクトキー。今回は `ALL` を想定   |
| `BACKLOG_PROJECT_ID`         | 任意 | ドキュメント追加APIに指定する数値のプロジェクトID。原則 `BACKLOG_PROJECT_KEY` から取得する。指定時は取得値との一致確認に使用する |
| `BACKLOG_PARENT_DOCUMENT_ID` | 必須 | 議事録登録先の親ドキュメントID            |
| `BACKLOG_ADD_LAST`           | 任意 | 兄弟ドキュメントの末尾に追加するか。原則 `true` |

`BACKLOG_NOTIFY` はBacklog Document APIの登録パラメータとして使用しないため、設定値に含めない。

APIキーはソースコードに直接記載しない。
環境変数、`.env` ファイル、または実行環境のシークレット管理機能で保持する。

---

### 5. プロジェクト確認

登録前に、`GET /api/v2/projects/{projectIdOrKey}` を使用して、登録先プロジェクトを確認する。

確認対象は以下とする。

| 確認項目         | 内容                                            |
| ------------ | --------------------------------------------- |
| プロジェクト実在確認   | `BACKLOG_PROJECT_KEY=ALL` のプロジェクトが存在すること      |
| projectKey確認 | レスポンスの `projectKey` が `ALL` であること             |
| projectId取得   | レスポンスの `id` を `BACKLOG_PROJECT_ID` として使用すること      |
| 参加権限確認       | API実行ユーザーが対象プロジェクトに参加していること                   |
| Document機能確認 | `useDocument` が `true` であること                  |
| archived確認    | レスポンスの `archived` が `false` であること             |

`BACKLOG_PROJECT_ID` は手入力を必須とせず、`BACKLOG_PROJECT_KEY` を基に `GET /api/v2/projects/{projectIdOrKey}` で取得した `id` を使用する。

キャッシュ済みの `BACKLOG_PROJECT_ID` がある場合は、取得した `id` と一致することを確認する。

`useDocument` が `false` の場合、Backlog Document APIによる登録処理は実行しない。

その場合は、Backlog側でドキュメント機能を有効化するか、登録先を見直す。

---

### 6. 登録先ドキュメントの特定方法

Backlogのドキュメントはツリー構造で管理されるため、議事録をどのドキュメント配下に登録するかを特定する必要がある。

初回設定時は、`GET /api/v2/documents/tree` を使用して、全社共通（ALL）のドキュメントツリーを取得する。

```text
GET /api/v2/documents/tree?projectIdOrKey=ALL
```

指定する主なパラメーターは以下とする。

| パラメーター           | 必須 | 内容                                  |
| ---------------- | -- | ----------------------------------- |
| `apiKey`         | 必須 | Backlog APIキー                       |
| `projectIdOrKey` | 必須 | プロジェクトIDまたはプロジェクトキー。今回は `ALL` を指定する |

確認対象は以下とする。

| 確認項目         | 内容                                   |
| ------------ | ------------------------------------ |
| 親ドキュメントID    | `BACKLOG_PARENT_DOCUMENT_ID` が存在すること |
| 親ドキュメント名     | 想定する登録先名と一致すること                      |
| 親ドキュメント階層    | 全社共通（ALL）の想定階層配下であること                |
| activeTree確認 | 登録先が `activeTree` 配下であること            |
| trashTree除外  | `trashTree` 配下のドキュメントを登録先にしないこと      |

運用時は、事前に設定した `BACKLOG_PARENT_DOCUMENT_ID` を使用する。
ただし、登録前に `GET /api/v2/documents/tree?projectIdOrKey=ALL` を実行し、親ドキュメントIDが `activeTree` 配下に存在し、想定する登録先であることを確認する。

### 7. 公開前確認フラグ

Backlog登録前に、公開前確認が完了していることを機械的に判定できるようにする。

公開前確認フラグは、Markdownファイルとは別に、確認情報ファイルとして保存する。

確認情報ファイル名は、Markdownファイル名と対応させる。

例：

```text
20260706_第1回システム設計×AI駆動開発勉強会_議事録.md
20260706_第1回システム設計×AI駆動開発勉強会_議事録.approval.json
```

ただし、Markdownファイルと同一フォルダに置くだけでは承認情報を書き換えられる可能性があるため、以下のいずれかの方法で承認情報の信頼性を担保する。

| 方式             | 方針                                             |
| -------------- | ---------------------------------------------- |
| 権限分離された保管場所    | approval.json を作成・更新できる場所を、承認者と運用管理者に限定する。承認内容は承認者が作成・更新し、運用管理者は保管場所・権限・監査ログの管理を担う。運用管理者がファイル管理上更新する場合も、`approved_by` には実際に公開前確認を行った承認者1名のIDを記録する  |
| Pull Request承認 | Markdownとapproval.jsonをレビュー対象にし、承認後のみ登録処理を実行する |
| 監査ログ           | approval.json の作成者、更新者、更新日時を記録する               |
| 電子署名           | 必要に応じてapproval.jsonに署名を付与する                    |

初期運用では、権限分離された保管場所またはPull Request承認のいずれかを採用する。

確認情報ファイルには以下を記録する。

```json
{
  "approved": true,
  "approved_by": "user001",
  "approved_at": "YYYY-MM-DDTHH:mm:ss+09:00",
  "markdown_file": "20260706_第1回システム設計×AI駆動開発勉強会_議事録.md",
  "markdown_sha256": "Markdown本文のSHA-256ハッシュ",
  "check_items": {
    "personal_information": true,
    "confidential_information": true,
    "internal_only_information": true,
    "uncertain_decisions": true,
    "todo_owner_due_date": true,
    "public_safe_expression": true
  }
}
```

`approved_by` には氏名ではなく、不変の承認者IDを記録する。
氏名は変更や重複の可能性があるため、承認者判定には使用しない。

Backlog登録処理では、以下を満たさない場合は登録しない。

* `approved` が `true` である
* `approved_by` が空でない
* `approved_by` が承認者リストに存在する
* `approved_by` の承認者が `active=true` である
* `approved_at` がISO 8601形式である
* `approved_at` が現在日時より未来でない
* ただし、端末時刻差を考慮し、現在日時より5分以内の未来日時は許容する
* Markdownファイル名と `markdown_file` が一致している
* Markdown本文のSHA-256と `markdown_sha256` が一致している
* `check_items` の全項目が `true` である

SHA-256の算出対象は、以下に統一する。

| 項目    | 仕様               |
| ----- | ---------------- |
| 文字コード | UTF-8            |
| BOM   | なし               |
| 改行コード | LF               |
| 算出対象  | Markdownファイル本文全体 |
| 除外対象  | なし               |

承認後変更検知の実装差を防ぐため、MarkdownファイルはUTF-8、BOMなし、LF改行に正規化したうえでSHA-256を算出する。

また、Backlogへ送信する `content` も、SHA-256算出時と同じ正規化済みMarkdown本文を使用する。

これにより、承認対象とBacklog登録対象の本文が一致するようにする。

### 7.1 承認者リスト

公開前確認を行える承認者は、承認者リストで管理する。

承認者リストは、設定ファイルまたはシークレット管理対象の設定として保持する。

例：

```json
{
  "approvers": [
    {
      "id": "user001",
      "name": "山田 太郎",
      "active": true
    },
    {
      "id": "user002",
      "name": "佐藤 花子",
      "active": true
    }
  ]
}
```

`id` は承認者判定に使用する不変IDとする。
`name` は表示用であり、承認判定には使用しない。

Backlog登録処理では、`approval.json` の `approved_by` が承認者リストに存在し、かつ `active=true` であることを確認する。

承認者の退職、異動、権限変更があった場合は、承認者リストを更新する。

承認者リストの管理者は、運用管理者とする。

### 8. 登録前チェック

Backlogへ登録する前に、以下のチェックを行う。

| チェック項目         | 内容                                                      |
| -------------- | ------------------------------------------------------- |
| APIキー確認        | `GET /api/v2/users/myself` で認証できるか                      |
| プロジェクト実在確認     | `GET /api/v2/projects/{projectIdOrKey}` で `ALL` が取得できるか |
| プロジェクト参加確認     | API実行ユーザーが `ALL` に参加しているか                               |
| Document機能確認   | `useDocument=true` であるか                                 |
| projectId確認    | `BACKLOG_PROJECT_ID` と取得した `id` が一致するか                  |
| 親ドキュメント確認      | `BACKLOG_PARENT_DOCUMENT_ID` が `activeTree` 配下に存在するか    |
| Markdownファイル確認 | 登録対象のMarkdownファイルが存在するか                                 |
| タイトル確認         | 議事録タイトルが空でないか                                           |
| 本文確認           | Markdown本文が空でないか                                        |
| 公開前確認フラグ       | approvalファイルが存在し、承認済みであるか                               |
| 承認日時確認         | `approved_at` がISO 8601形式で、未来日時でないか                     |
| 承認者確認          | `approved_by` が承認者として認められたユーザーであるか                      |
| チェック項目確認       | `check_items` の全項目が `true` であるか                         |
| ハッシュ確認         | 承認時のMarkdownハッシュと現在のMarkdownハッシュが一致するか                  |
| 同名ドキュメント確認     | 同じタイトルのドキュメントが存在しないか                                    |

公開前確認が完了していない場合、承認後にMarkdown本文が変更されている場合、または `check_items` に `false` が含まれる場合、Backlog登録処理は実行しない。

### 9. 重複登録防止

同じ議事録を重複登録しないため、登録前にドキュメント一覧取得APIおよびドキュメントツリー取得APIで同名ドキュメントを確認する。

本運用では、重複判定のスコープを以下のように分ける。

| 確認範囲                                | 判定          | 扱い                     |
| ----------------------------------- | ----------- | ---------------------- |
| プロジェクト全体                            | 同名タイトルが存在する | 警告として扱う                |
| `BACKLOG_PARENT_DOCUMENT_ID` の直下    | 同名タイトルが存在する | 登録停止                   |
| `BACKLOG_PARENT_DOCUMENT_ID` の孫階層以降 | 初期運用では確認対象外 | 年月別階層運用に変更する場合は再帰確認に変更 |

`GET /api/v2/documents` による検索では、`keyword` 検索結果をそのまま信用せず、取得結果の `title` と登録予定タイトルを完全一致で照合する。

検索条件は以下とする。

| パラメーター        | 内容                 |
| ------------- | ------------------ |
| `projectId[]` | 全社共通（ALL）のプロジェクトID |
| `keyword`     | 登録予定の議事録タイトル       |
| `sort`        | `updated`          |
| `order`       | `desc`             |
| `offset`      | `0` からページング        |
| `count`       | `100`              |

`count=100` で取得し、100件取得された場合は `offset` を増やして次ページを取得する。
取得件数が100件未満になるまでページングし、全件を確認する。

親ドキュメント直下の重複確認は、`GET /api/v2/documents/tree?projectIdOrKey=ALL` で取得したツリーを使用する。

ドキュメントツリーでは、`BACKLOG_PARENT_DOCUMENT_ID` に該当するノードを特定し、その `children` 配列を確認する。

親直下重複確認では、`children[].name` と登録予定タイトルを比較する。

以下の場合は登録を停止する。

* `BACKLOG_PARENT_DOCUMENT_ID` の `children` 直下に、登録予定タイトルと同じ `name` のドキュメントが存在する
* 同名確認中にAPIエラーが発生し、重複有無を確認できない
* ツリー上で `BACKLOG_PARENT_DOCUMENT_ID` が見つからない
* 対象親ドキュメントが `trashTree` 配下に存在する

プロジェクト全体に同名タイトルが存在するが、指定親ドキュメント直下には存在しない場合は、警告ログを出力したうえで登録を継続可能とする。

ただし、運用上、プロジェクト全体で同名議事録を禁止する場合は、プロジェクト全体一致も登録停止条件に変更する。

再実行時も、登録前に必ず同名ドキュメント検索を行う。

### 9.1 同時実行時の排他制御

ドキュメント一覧取得からドキュメント追加までの間に、別プロセスが同名ドキュメントを登録すると、重複登録が発生する可能性がある。

そのため、Backlog登録処理では、ジョブ単位の排他ロックを行う。

排他ロックは、単一プロセス内のメモリロックではなく、複数実行環境から共有できる方式を使用する。

初期運用では、以下のいずれかを採用する。

| 実行環境               | 排他ロック方式                   |
| ------------------ | ------------------------- |
| 1台のWindows PCで手動実行 | 共有フォルダ上のロックファイル           |
| CI/CDで実行           | CIの concurrency group     |
| 複数サーバーで実行          | Redis、DB、または共有ストレージ上のロック  |
| 実装方式が未確定の場合        | ファイルロックを暫定採用し、運用環境決定後に見直す |

排他ロックの単位は以下とする。

| 項目       | 内容                                                                |
| -------- | ----------------------------------------------------------------- |
| ロック対象    | 登録予定タイトル                                                          |
| ロックキー    | `BACKLOG_PROJECT_KEY + BACKLOG_PARENT_DOCUMENT_ID + 正規化済み議事録タイトル` |
| ロックタイミング | Backlog登録前チェック開始時                                                 |
| ロック解除    | 登録成功、登録失敗、処理中断時                                                   |
| ロック保持時間  | 原則30分。異常終了時は期限切れ扱い                                                |
| ロック取得失敗時 | 登録処理を実行せず、別ジョブ実行中として停止する                                          |

排他ロック取得後も、POST登録直前に再度同名ドキュメント検索を行う。

POST登録後にも、同名タイトルのドキュメントが複数存在しないか確認する。
同名ドキュメントが複数検出された場合は、登録成功扱いにせず、重複登録アラートとしてログに記録し、担当者に確認を求める。

---

### 9.2 タイトル正規化

同名確認および登録時のタイトルは、以下のルールで正規化する。

| 項目      | 仕様          |
| ------- | ----------- |
| Unicode正規化 | NFC         |
| 前後空白    | 削除          |
| 連続空白    | 1つに統一       |
| 改行・タブ   | 半角スペースに置換   |
| 不可視文字   | 削除          |
| 比較対象    | 正規化後タイトル    |

登録に使用するタイトルも、同名確認で使用した正規化済みタイトルと同じものを使用する。

### 10. ドキュメント登録仕様

Backlogへの登録には、ドキュメント追加APIを使用する。

```text
POST /api/v2/documents
```

リクエスト形式は以下とする。

| 項目           | 内容                                  |
| ------------ | ----------------------------------- |
| HTTPメソッド     | `POST`                              |
| エンドポイント      | `/api/v2/documents`                 |
| 認証           | クエリパラメーター `apiKey`                  |
| Content-Type | `application/x-www-form-urlencoded` |

送信する主なパラメーターは以下とする。

| パラメーター      | API仕様上の必須 | 本運用上の必須 | 設定値                          |
| ----------- | --------- | ------- | ---------------------------- |
| `projectId` | 必須        | 必須      | `BACKLOG_PROJECT_ID`         |
| `title`     | 任意        | 必須      | 議事録タイトル                      |
| `content`   | 任意        | 必須      | Markdown形式の議事録本文             |
| `emoji`     | 任意        | 任意      | 空文字または任意の絵文字                 |
| `parentId`  | 任意        | 必須      | `BACKLOG_PARENT_DOCUMENT_ID` |
| `addLast`   | 任意        | 任意      | 原則 `true`                    |

`title`、`content`、`parentId` はAPI仕様上は任意だが、本運用では議事録としての登録先・タイトル・本文を保証するため必須扱いとする。

`BACKLOG_NOTIFY` または `notify` に相当する通知制御パラメータは使用しない。

---

### 11. 登録後確認

Backlog登録後は、APIレスポンス、ドキュメント取得API、ドキュメントツリー取得APIで登録結果を確認する。

確認項目は以下とする。

| 確認項目        | 内容                                                                                                                      |
| ----------- | ----------------------------------------------------------------------------------------------------------------------- |
| ドキュメントID    | APIレスポンスにIDが含まれているか                                                                                                     |
| プロジェクトID    | 全社共通（ALL）のprojectIdと一致しているか                                                                                             |
| タイトル        | 想定タイトルと一致しているか                                                                                                          |
| 本文          | ドキュメント取得APIの本文が空でなく、可能であれば正規化済みMarkdown本文と一致すること                                                                       |
| 親ドキュメント配下確認 | `GET /api/v2/documents/tree?projectIdOrKey=ALL` を再取得し、追加されたドキュメントIDが `BACKLOG_PARENT_DOCUMENT_ID` の `children` 直下に存在するか |
| 同名重複確認      | 登録後に同名タイトルのドキュメントが複数存在しないか                                                                                              |
| 登録日時        | 登録日時が取得できるか                                                                                                             |

ドキュメント取得APIのレスポンスだけで、親ドキュメント配下であることを確認しない。

親配下確認は、登録後にドキュメントツリーを再取得し、作成されたドキュメントIDが想定親ドキュメントの `children` 直下に存在することを確認する。

登録後に同名タイトルのドキュメントが複数存在する場合は、同時実行などによる重複登録の可能性があるため、重複登録アラートとして扱う。

登録成功時は、以下をログに出力する。

```text
Backlogドキュメント登録が完了しました。
タイトル: 20260706_第1回システム設計×AI駆動開発勉強会_議事録
ドキュメントID: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
登録先: 全社共通（ALL）ドキュメント配下
```

### 12. 誤登録時の削除・復旧手順

誤登録が判明した場合は、即時自動削除は行わない。

削除は、管理者またはプロジェクト管理者が内容を確認したうえで実施する。

削除対象の確認項目は以下とする。

| 確認項目     | 内容                       |
| -------- | ------------------------ |
| ドキュメントID | 削除対象のIDが登録ログと一致しているか     |
| タイトル     | 削除対象のタイトルが想定と一致しているか     |
| 登録先      | 全社共通（ALL）の対象ドキュメント配下であるか |
| 登録日時     | 誤登録対象の登録日時と一致しているか       |
| 削除判断者    | 管理者またはプロジェクト管理者であるか      |

削除する場合は、以下のAPIを使用する。

```text
DELETE /api/v2/documents/{documentId}
```

削除実行後は、以下をログに残す。

* 削除日時
* 削除対象ドキュメントID
* 削除対象タイトル
* 削除実行者
* 削除理由
* 削除前確認者
* 削除APIの実行結果

削除後に再登録が必要な場合は、修正済みMarkdownを再承認し、新しい承認情報ファイルを作成したうえで登録する。

### 13. エラー時の扱い

Backlog通信時に発生する主なエラーと対応は以下とする。

| エラー                   | 想定原因                   | 対応                              |
| --------------------- | ---------------------- | ------------------------------- |
| 401 Unauthorized      | APIキー不正・期限切れ           | APIキーを確認する                      |
| 403 Forbidden         | 権限不足                   | API実行ユーザーがALLプロジェクトに参加しているか確認する |
| 404 Not Found         | URL、プロジェクト、親ドキュメントID誤り | 設定値を確認する                        |
| 429 Too Many Requests | API実行回数制限              | 13.1の方針で再試行する                   |
| 5xx Server Error      | Backlog側障害または一時エラー     | リトライ後、失敗する場合は手動登録へ切り替える         |
| 同名ドキュメントあり            | 重複登録の可能性               | 自動上書きせず停止する                     |
| Markdown本文空           | 議事録生成失敗                | Backlog登録せず停止する                 |
| Document機能無効          | `useDocument=false`    | Backlog登録せず停止する                 |
| 承認情報不一致               | 承認後にMarkdownが変更された可能性  | Backlog登録せず再承認を求める              |

#### 13.1 429発生時のリトライ方針

429 Too Many Requests が発生した場合は、以下の優先順位で待機時間を決定する。

1. `Retry-After` ヘッダーがある場合は、その値を最優先する
2. `Retry-After` がなく、`X-RateLimit-Reset` がある場合は、現在時刻と `X-RateLimit-Reset` の差分を待機時間とする
3. どちらも取得できない場合は、補助APIとして `GET /api/v2/rateLimit` を実行し、該当するリクエスト種別の残数とリセット時刻を確認する
4. それでも判断できない場合は、固定待機時間で再試行する

POST登録は更新系リクエストとして扱い、`update` 区分の残数とリセット時刻を参照する。

固定待機時間は以下とする。

| 回数  | 待機時間 |
| --- | ---- |
| 1回目 | 5秒   |
| 2回目 | 15秒  |
| 3回目 | 30秒  |
| 4回目 | 60秒  |

最大4回まで再試行する。

POST登録処理を再試行する前には、必ず同名ドキュメント検索を再実行する。
これは、直前のPOSTがタイムアウトしたものの、Backlog側では登録が成功している可能性があるためである。

最大回数を超えても失敗する場合は処理を停止し、手動登録または時間を置いた再実行に切り替える。

### 14. ログ出力

Backlog連携処理では、以下をログとして保存する。

| ログ項目         | 内容                |
| ------------ | ----------------- |
| 実行日時         | 処理開始・終了日時         |
| 登録対象ファイル     | Markdownファイル名     |
| 承認情報ファイル     | approvalファイル名     |
| 承認者          | `approved_by`     |
| 承認日時         | `approved_at`     |
| Markdownハッシュ | `markdown_sha256` |
| 登録タイトル       | Backlogドキュメントタイトル |
| 登録先プロジェクト    | 全社共通（ALL）         |
| projectId    | 登録先プロジェクトID       |
| 親ドキュメントID    | 登録先ID             |
| 実行結果         | 成功・失敗             |
| エラー内容        | 失敗時のエラーメッセージ      |
| ドキュメントID     | 登録成功時のID          |
| 削除実行情報       | 誤登録削除時のみ記録        |

ただし、以下はログに出力しない。

* APIキー
* `apiKey` を含むURL
* クエリ文字列全体
* 議事録本文全文
* 承認前のMarkdown全文

エラーログにURLや例外メッセージを出力する場合は、必ず `apiKey` をマスクする。

マスク例：

```text
https://example.backlog.jp/api/v2/documents?apiKey=********
```

例外オブジェクトやHTTPクライアントのデバッグログをそのまま出力しない。
ログ出力前に、URL、クエリパラメーター、ヘッダー、本文のうち、機密情報を含む可能性がある値をマスクする。

### 15. セキュリティ要件

Backlog APIキーは、Backlog上でドキュメント登録権限を持つユーザーに紐づく認証情報である。

そのため、以下を守る。

* APIキーをソースコードに直接書かない
* APIキーをGit管理対象に含めない
* `.env` ファイルを使用する場合は `.gitignore` に追加する
* APIキーをログに出力しない
* APIキーをチャットやメール本文に貼らない
* API実行ユーザーは、必要最小限の権限にする
* 誤登録削除を行うAPIキーは、管理者またはプロジェクト管理者の権限を持つものに限定する
* 通常登録用APIキーと削除用APIキーを分けることを検討する
* 不要になったAPIキーは無効化する

### 16. Backlog登録処理の流れ

Backlog登録処理の全体フローは以下とする。

```text
Markdown議事録生成
  ↓
公開前確認
  ↓
approval.json 作成
  ↓
Markdownハッシュ確認
  ↓
APIキー確認
  ↓
GET /api/v2/projects/{projectIdOrKey} でプロジェクト確認
  ↓
useDocument=true / archived=false 確認
  ↓
登録先親ドキュメント確認
  ↓
排他ロック取得
  ↓
同名ドキュメント検索
  ↓
POST直前の同名ドキュメント再検索
  ↓
同名なし
  ↓
POST /api/v2/documents で登録
  ↓
登録後のツリー再取得
  ↓
親配下登録確認
  ↓
登録後の同名重複確認
  ↓
ログ出力
  ↓
排他ロック解除
```

同名ドキュメントが存在する場合は、以下の流れとする。

```text
同名ドキュメント検索
  ↓
同名あり
  ↓
自動登録停止
  ↓
担当者へ確認
  ↓
登録中止または手動判断
```

誤登録が発生した場合は、以下の流れとする。

```text
誤登録判明
  ↓
削除対象ドキュメントID確認
  ↓
管理者またはプロジェクト管理者が削除可否判断
  ↓
DELETE /api/v2/documents/{documentId}
  ↓
削除結果をログ出力
  ↓
必要に応じて修正版を再承認・再登録
```
