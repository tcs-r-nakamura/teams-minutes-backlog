## 音声テキスト化方式の選定

### 1. 採用方式

本件では、Teams録画から文字起こしを行う手段として、whisper.cppを採用する。

Teamsトランスクリプトは使用せず、Teams録画ファイルから音声を抽出し、ローカル環境で whisper.cpp を実行して文字起こしを行う。

### 2. 採用理由

whisper.cpp を採用する理由は以下のとおりである。

* 外部API費用や追加クラウド課金なしで利用できる
* ローカルPC上で処理できる
* Teams録画や音声ファイルを社外サービスへアップロードしなくてよい
* コマンドラインで実行できるため、手順化・自動化しやすい
* Windows環境でも利用できる
* OpenAI Whisper系のモデルを利用でき、日本語の文字起こしにも対応できる
* 1時間未満の会議録画であれば、今回の用途に適している
* Teams側のトランスクリプト設定や会議中の文字起こし開始操作に依存しない
* 録画ファイルさえ入手できれば、後から同じ手順で文字起こしを実行できる

### 3. Teamsトランスクリプトを採用しない理由

本件では、Teamsトランスクリプトは採用しない。

理由は以下のとおりである。

* 今回の依頼が「録画を基に議事録を作成する」ものであるため
* Teamsトランスクリプトは、会議設定や会議中の文字起こし開始有無に依存するため
* 録画ファイルが保存されていれば、会議後に同じ手順で処理できるため、運用の起点として扱いやすいため
* 今後の運用でも、録画ファイルを入力とする方が手順を統一しやすいため
* Teamsトランスクリプトの有無によって手順が分岐することを避けるため

### 4. 処理方針

処理の流れは以下とする。

```text
Teams録画ファイル
  ↓
FFmpegで16kHz・モノラル・16bit PCM WAVへ変換
  ↓
whisper.cppで文字起こし
  ↓
文字起こし結果をtxt / srt / vtt形式で出力
  ↓
AIで議事録案を作成
  ↓
Markdown形式に整形
```

## whisper.cpp入力音声フォーマット定義

### 1. 基本方針

本件では、Teams録画ファイルを直接 whisper.cpp に渡さない。

Teams録画ファイルからFFmpegで音声を抽出し、whisper.cpp が処理できる形式に変換したうえで文字起こしを行う。

本運用では、whisper.cpp に渡す音声ファイルを **16kHz・モノラル・16bit PCM WAV** に統一する。

### 2. 入力音声フォーマット

whisper.cpp に渡す音声ファイルは、以下の形式に統一する。

| 項目        | 設定値                             |
| --------- | ------------------------------- |
| ファイル形式    | WAV                             |
| サンプリングレート | 16kHz                           |
| チャンネル数    | モノラル                            |
| 音声コーデック   | PCM signed 16-bit little-endian |
| 拡張子       | `.wav`                          |

### 3. 変換コマンド

Teams録画ファイルから whisper.cpp 用の音声ファイルを作成する場合は、以下のFFmpegコマンドを使用する。

```bash
ffmpeg -i 録画.mp4 -ar 16000 -ac 1 -c:a pcm_s16le audio.wav
```

Windows環境でファイルパスに空白が含まれる場合は、以下のようにダブルクォーテーションで囲む。

```bash
ffmpeg -i "C:\path\to\録画.mp4" -ar 16000 -ac 1 -c:a pcm_s16le "C:\path\to\audio.wav"
```

### 4. コマンドの意味

| オプション            | 意味                                   |
| ---------------- | ------------------------------------ |
| `-i 録画.mp4`      | 入力するTeams録画ファイル                      |
| `-ar 16000`      | サンプリングレートを16kHzに変換する                 |
| `-ac 1`          | 音声チャンネルをモノラルに変換する                    |
| `-c:a pcm_s16le` | 音声コーデックを16bit PCM little-endianに変換する |
| `audio.wav`      | whisper.cpp に渡す出力音声ファイル              |

### 5. 音声トラック確認

変換前に、録画ファイルに音声トラックが含まれているか確認する。

```bash
ffprobe 録画.mp4
```

音声トラックが存在しない場合は、文字起こし処理を実行せず、録画ファイルの取得元を確認する。

複数の音声トラックが存在する場合は、使用する音声トラックを確認し、必要に応じて `-map` オプションで対象トラックを指定する。

例：

```bash
ffmpeg -i 録画.mp4 -map 0:a:0 -ar 16000 -ac 1 -c:a pcm_s16le audio.wav
```

### 6. 変換後WAVの確認

変換後、出力されたWAVファイルが想定形式になっているか確認する。

```bash
ffprobe audio.wav
```

確認観点は以下とする。

* WAV形式であること
* サンプリングレートが16kHzであること
* モノラルであること
* 16bit PCM形式であること

## whisper.cpp実行定義

### 1. 実行コマンド

変換後の音声ファイルを使用して、whisper.cppで文字起こしを実行する。

```bash
whisper-cli -m models/ggml-medium.bin -f audio.wav -l ja -otxt -osrt -ovtt
```

環境によっては、実行バイナリ名が `whisper-cli` ではなく `main` の場合がある。

その場合は、以下のように読み替える。

```bash
main -m models/ggml-medium.bin -f audio.wav -l ja -otxt -osrt -ovtt
```

実際のコマンド名、モデルファイル名、モデルの配置場所は、インストールした whisper.cpp のバージョンおよび環境に合わせて調整する。

### 2. モデルファイル

初期設定では `medium` 相当のモデルを使用する。

ただし、環境によっては量子化モデルを使用する場合がある。

例：

```text
ggml-medium.bin
ggml-medium-q5_0.bin
ggml-large-v3.bin
ggml-large-v3-q5_0.bin
```

モデルファイルは事前に取得し、whisper.cpp から読み込める場所に配置しておく。

### 3. 使用モデル

文字起こし精度を重視するため、Whisperモデルは `medium` 以上を基本とする。

処理時間が長すぎる場合は `small` に下げることを検討する。

`large` は精度重視時に使用するが、CPUのみの環境では処理時間やメモリ負荷が大きくなる可能性があるため、高性能PCまたはGPUが利用できる場合の選択肢とする。

| モデル    | 方針                            |
| ------ | ----------------------------- |
| tiny   | 使用しない                         |
| base   | 使用しない                         |
| small  | 検証用、または処理時間短縮時に使用             |
| medium | 初期採用                          |
| large  | 精度重視時に使用。ただし高性能PCまたはGPU利用時を推奨 |

## 出力形式

### 1. 出力ファイル

whisper.cppの出力は、以下を基本とする。

| 出力形式   | 用途           | 方針                         |
| ------ | ------------ | -------------------------- |
| `.txt` | AI議事録生成用     | 必須                         |
| `.srt` | 時刻付き確認用      | `.srt` または `.vtt` のいずれかを必須 |
| `.vtt` | 時刻付き確認用・字幕形式 | `.srt` または `.vtt` のいずれかを必須 |

議事録生成には `.txt` を使用する。

ただし、後から発言箇所を確認できるようにするため、時刻付き出力として `.srt` または `.vtt` のいずれかを保存する。
可能であれば、`.srt` と `.vtt` の両方を保存する。

### 2. `.txt` だけにしない理由

`.txt` だけでは、議事録作成後に元発言の位置を追跡しにくい。

そのため、議事録生成には `.txt` を使用しつつ、確認用として `.srt` または `.vtt` を保存する。

## 話者分離の扱い

### 1. 基本方針

本件では、whisper.cpp単体による話者分離は行わない。

whisper.cppの文字起こし結果は、基本的に発言内容のテキスト化を目的とし、話者ごとの厳密な区別は対象外とする。

### 2. 話者名の扱い

議事録本文では、発言者ごとの逐語録を作成するのではなく、会議の要点、決定事項、TODO、確認事項を整理する。

そのため、話者名の完全な特定は必須としない。

ただし、TODOの担当者、決定事項に関わる責任者、確認依頼先など、議事録上重要な人物名については手動確認対象とする。

### 3. Teams参加者名との対応

whisper.cppの出力結果とTeams参加者名の対応付けは、自動では行わない。

必要な場合は、担当者が手動で確認する。

### 4. 受け入れ条件

話者分離に関する受け入れ条件は以下とする。

* whisper.cpp単体での話者分離は対象外であることが明記されている
* 発言者ごとの完全な発言録は作成対象外である
* 担当者、期限、決定事項に関わる人物名は手動確認対象である
* Teams参加者名との対応付けは自動化対象外である

## 処理時間・ハードウェア前提

### 1. 基本方針

whisper.cppの処理時間は、録画時間、使用モデル、PC性能、CPU/GPU利用有無によって変動する。

そのため、初回運用では処理時間を測定し、以後の運用目安として記録する。

### 2. 推奨環境

初期運用では、以下を目安とする。

| 項目          | 目安                          |
| ----------- | --------------------------- |
| OS          | Windows 10 / Windows 11     |
| CPU         | 64bit CPU                   |
| メモリ         | 16GB以上推奨                    |
| ストレージ       | 録画ファイル、WAV、出力ファイルを保存できる空き容量 |
| GPU         | 必須ではない。ただし高速化したい場合は利用を検討    |
| FFmpeg      | コマンド実行可能であること               |
| whisper.cpp | ビルド済み、または実行可能ファイルが用意されていること |
| Whisperモデル  | 使用するモデルファイルが取得済みであること       |

### 3. 処理時間の目安

1時間未満の動画を対象とする。

初期運用では、文字起こし処理が 録画時間の3倍以内で完了することを目標とする。

例：

| 録画時間  | 目標処理時間 |
| ----- | ------ |
| 30分   | 90分以内  |
| 60分未満 | 180分以内 |

この目標を大きく超える場合は、以下を検討する。

* `medium` から `small` へ変更する
* 量子化モデルを使用する
* GPU利用環境へ変更する
* 処理対象音声を分割する
* ノイズ除去や無音区間削除を検討する

### 4. largeモデルの扱い

`large` モデルは精度重視時に使用する。

ただし、CPUのみの環境では処理時間が長くなり、メモリ使用量も増える可能性がある。

そのため、`large` は以下の場合に限定して使用する。

* 高性能PCを利用できる場合
* GPUを利用できる場合
* 処理時間よりも正確性を優先する場合
* `medium` で精度不足が確認された場合

## 誤認識しやすい項目

whisper.cppによる文字起こし結果は、議事録作成の元データとして使用する。

ただし、文字起こし結果を完全に正確な発言録として扱わない。

特に以下は誤認識の可能性があるため、議事録作成時に確認対象とする。

* 人名
* 会社名
* 部署名
* プロジェクト名
* 製品名
* サービス名
* 社内用語
* 略語
* 英数字混じりの用語
* 同音異義語
* 日付
* 数字
* 金額
* 担当者
* 期限
* 決定事項
* TODO

## エラー時の扱い

### 1. 音声変換エラー

FFmpegで音声変換に失敗した場合は、以下を確認する。

1. 録画ファイルが存在するか
2. ファイルパスに誤りがないか
3. ファイルパスに空白がある場合、ダブルクォーテーションで囲んでいるか
4. 録画ファイルに音声トラックが含まれているか
5. FFmpegがインストールされ、コマンド実行可能であるか
6. 出力先フォルダに書き込み権限があるか

### 2. 音声トラックがない場合

録画ファイルに音声トラックが存在しない場合は、文字起こし処理を実行しない。

その場合は、録画ファイルの取得元、Teams録画設定、別ファイルの有無を確認する。

### 3. 複数音声トラックがある場合

録画ファイルに複数の音声トラックが存在する場合は、対象トラックを確認する。

必要に応じて、FFmpegの `-map` オプションで使用する音声トラックを指定する。

### 4. whisper.cpp実行エラー

whisper.cpp 実行時にエラーが発生した場合は、以下を確認する。

1. whisper.cppの実行ファイルが存在するか
2. コマンド名が環境に合っているか
3. モデルファイルが取得済みであるか
4. モデルファイルのパスが正しいか
5. 入力音声ファイルがWAV形式であるか
6. サンプリングレートが16kHzであるか
7. チャンネル数がモノラルであるか
8. コーデックが16bit PCMであるか
9. メモリ不足が発生していないか

形式が不一致の場合は、再度以下のコマンドで変換してから実行する。

```bash
ffmpeg -i 録画.mp4 -ar 16000 -ac 1 -c:a pcm_s16le audio.wav
```

## 受け入れ条件

本方式の受け入れ条件は以下とする。

### 1. 入力・変換

* Teams録画ファイルから音声を抽出できる
* FFmpegで16kHz・モノラル・16bit PCM WAVへ変換できる
* 変換後WAVを `ffprobe` で確認できる
* 音声トラックがない場合は処理を停止できる
* 複数音声トラックがある場合の扱いが明記されている

### 2. whisper.cpp実行

* whisper.cppの実行ファイルが用意されている
* 環境に応じて `whisper-cli` または `main` が実行できる
* 使用するWhisperモデルファイルが取得済みである
* モデルファイルの配置場所が定義されている
* whisper.cppで文字起こしを実行できる
* whisper.cppで音声ファイルの読み込みエラーが発生しない

### 3. 出力

* 文字起こし結果を `.txt` 形式で出力できる
* 時刻付き確認用として `.srt` または `.vtt` のいずれかを出力できる
* 可能であれば `.srt` と `.vtt` の両方を保存できる
* `.txt` をAI議事録生成用の元データとして使用できる
* `.srt` または `.vtt` を発言箇所確認用として保存できる

### 4. 処理時間・環境

* 1時間未満の動画を処理できる
* 初期運用では録画時間の3倍以内で処理完了することを目標とする
* 処理時間を実行ログに記録できる
* 使用PCのCPU、メモリ、使用モデルを記録できる
* 処理時間が目標を大きく超える場合の対応方針が定義されている

### 5. 話者分離

* whisper.cpp単体での話者分離は対象外である
* 発言者ごとの完全な発言録は作成対象外である
* Teams参加者名との自動対応付けは対象外である
* 担当者、期限、決定事項に関わる人物名は手動確認対象である

### 6. セキュリティ・費用

* 外部API費用や追加クラウド課金なしで実行できる
* 社外サービスへ録画・音声をアップロードせずに処理できる
* ローカル環境で処理が完結する
* 後続のAI議事録生成処理へ連携できる
