## 1. 目的・ゴール

- 議事録作成の「AI草案生成（現状は人がChatGPTへコピペ）」をAPI化して自動化する（＝現行手順の③を自動化）。
- 将来的に、トリガ発火後の処理（文字起こし・草案生成・Backlog反映）をVPS上で完結させ、各自PCでスクリプトを回さなくてよい社内サービスにする。
- 公開前チェック（人名・数値・日付・機微情報の確認）は引き続き人が実施する。**VPSが自動で作るのは「ドラフト」まで**で、Backlogへの本登録は人の承認後に限定する（8章）。

### スコープ外（今回やらないこと）
- 公開前チェックの自動化（意図的に人手を残す）。
- Cybozuからの録画自動取得（公式APIが無いため、当面は人手ダウンロードを前提）。
- 未チェック草案の全社共有領域への自動公開。

## 2. 現状（As-Is）と課題

### できていること（PC版）
- 録画取り込み → 音声抽出（FFmpeg） → 文字起こし（whisper.cpp、PC内完結）
- Teamsトランスクリプト（.vtt）併用：話者名つきテキストとwhisperの2ソースをAIに併記、登壇者を自動セット
- 会議番号（Backlog自動採番）・開催日・録画リンクの自動決定
- AIプロンプト自動生成 → （人がAIへ貼付・.md取得） → Backlog登録

### 課題（今回対処したい）
- ③のAI草案生成が手作業（ブラウザでコピペ・.mdダウンロード）。
- 文字起こしがPCのCPU依存で重い／低スペックPCで遅い。
- 各自PCにツール一式を配布・維持する必要がある。

## 3. 利用コンポーネント

| コンポーネント | 役割 | 備考 |
|---|---|---|
| Cybozu Office | 録画（.mp4）・トランスクリプト（.vtt）の保管 | 公式ダウンロードAPIなし＝取得は人手 |
| VPS（NixOS, AI-OCR用） | サーバー側処理の実行基盤 | `~/projects/minutes/` 配下に自己完結で構築。グローバル依存は足さない |
| さくらのAI Engine | 文字起こし（Whisper）・草案生成（LLM） | OpenAI互換API・国内完結・無料枠。承認済み |
| Backlog | 議事録ドキュメントの登録先 | Document API。追加可・削除不可アカウント |

### さくらAI Engine 仕様（設計の前提）
- API：OpenAI互換 Chat Completions／Responses、Anthropic互換 Messages。
- 認証：APIアカウントトークン（Bearer。環境変数 `SAKURA_AI_TOKEN` で管理、ファイルに置かない）。
- モデル：文字起こし `whisper-large-v3-turbo`／LLM `gpt-oss-120b`・`llm-jp-3.1-8x13b`・`PLaMo 2.0-31B`・`cotomi v3` 他。
- 無料枠：Chat **3,000回/月**、音声文字起こし **50回/月**、Embeddings 1万/月。
- データ取扱い：**国内クラウド完結・学習利用なし・承認済みAPI（さくらAI Engine）へのみ送信**（国外／未承認AIへは送信しない）。※音声・文字起こしはこの承認済みAPIへは送信される（社内VPS内で完結はしない）点に留意。

#### 文字起こしAPIの具体仕様（出典：さくら公式ナレッジ／マニュアル）
- エンドポイント：`POST https://api.ai.sakura.ad.jp/v1/audio/transcriptions`（OpenAI互換）
- パラメータ：`file`（multipart）＋`model=whisper-large-v3-turbo`
- 対応：mp3/wav/m4a/mp4 等。**1リクエストの上限は「最長30分 または 30MB」**（内部で30秒ごとに分割・結合）
- レスポンス：`{"text":"...","model":"whisper-large-v3-turbo"}` の**テキストのみ**（話者名・タイムスタンプは返らない）

> ★設計に効く制約：当会議は**45分超**があり30分上限を超える。
> - 送信するのは**動画(mp4)ではなく、ffmpegで抽出・mp3圧縮した音声**（例：元動画205MB → 音声mp3は約10〜20MB）。動画そのものは送らない。
> - それでも「30**分**上限」があるため、音声は**分割**して複数回リクエストし、`text`を順に結合する。実装要件は「3.1」に定義する。
> - 話者割当はTeams .vttを主とし、Whisperは聞き取り・固有名詞の**補正**に使う（次項）。消費回数は「1会議＝分割数（例：2）」で数える（10章）。

#### 話者割当の方針（VTT主・Whisper補正）（#1対応）
Whisperのレスポンスは`text`のみで話者・タイムスタンプが無いため、Whisper単独では「誰が」を機械的に割り当てられない。したがって：
- **Teams .vttの話者付き発話を骨格（＝誰が）**とし、**Whisper全文で聞き取り誤り・固有名詞・言い回しを補正（＝何を）**する。
- .vttが欠落、または.vttとWhisperの内容が大幅に食い違う場合は、話者推定を断定せず**人手確認（要確認）に回す**。
- 「Whisperを話者の正」とは扱わない。

### 3.1 文字起こしの分割・結合の実装要件（#2・#3・#5対応）
長音声対応は「分割できる」だけでなく、サイズ保証・分割位置・重複除去・順序結合まで定義する。

1. **サイズ/長さの保証（#2）**：`ffprobe`で音声のduration/sizeを検査 → 目標**25分程度**で分割 → 各チャンクをエンコード後に **duration < 30:00 かつ size < 30MB** を検証 → 超過時は再分割または低ビットレート再エンコード → **API送信前に満たさなければハードfail**（黙って送らない）。mp3ビットレートは音声認識に十分な範囲（例：モノラル 48〜64kbps）で固定。
2. **分割位置＝無音（#5）**：`ffmpeg -af silencedetect=noise=-35dB:d=0.5` で無音区間を検出し、目標時刻付近（探索範囲例：±3分）の**無音区間の中央**で切る。会議音声は閾値次第で過検出/未検出が起きるため、**候補が無い場合は固定時刻で分割し「要確認」フラグ**を立てる。境界に**数秒のオーバーラップ**を持たせる。
3. **オーバーラップ重複除去（#3）**：レスポンスはtextのみで時間情報が無いため、`chunk_i`末尾N文字と`chunk_{i+1}`先頭N文字の**文字列類似度で最長重複を検出し、一定閾値以上のときだけ削除**する。低信頼のときは**削除せず境界マーカー（例：`〔chunk境界〕`）を残す**（発言を誤って削らない安全側）。テストケース（重複あり/なし/部分一致）を用意する。
4. **順序結合**：`chunk_index` 昇順で結合。**全チャンク成功時のみ**結合・後続処理へ進む（9章のチャンク単位状態管理）。

> 枠の含意（確定）：勉強会は**月最大5回**のため、文字起こし50回/月には十分収まる（約10倍の余裕）。→ **文字起こしは毎回さくらのWhisper（`whisper-large-v3-turbo`）を使用**する。話者名は Teams .vtt を併用（Whisperは話者分離しないため）。消費は少ないが、失敗再実行・長時間分割での増加に備え、簡易な利用台帳だけ持つ（10章）。この方針により**ローカルのwhisper.cpp（約1.5GBモデル・重いCPU処理・未署名exeのAV警告）が不要**になり構成が単純化する。

## 5. VPS上の構成（`~/projects/minutes/`）

制約：ツール導入は**このフォルダ内のNix devenv/flakeで自己完結**させ、グローバル（システム全体）への依存追加はしない。

```
~/projects/minutes/
  flake.nix / flake.lock   # 依存の定義と固定（python3, ffmpeg, curl 等をこの中だけで）
  devenv.nix / .envrc      # 開発シェル（direnv連携は任意）
  src/
    build_prompt.py        # 2ソースのプロンプト組立（PC版ロジックの移植）
    parse_vtt.py           # Teams .vtt 解析（話者正規化・speaker_aliases）
    transcribe.py          # 音声を30分/30MB未満に分割→さくらWhisper呼び出し→text結合（ffmpeg利用）
    draft.py               # さくらAI Engine Chat Completions 呼び出し（③）
    register.py            # Backlog Document API 登録（承認後のみ）＋通知
    jobstore.py            # ジョブ/状態・冪等性（9章）
    quota.py               # 無料枠の消費記録・しきい値（10章）
    server.py              # 入口（FastAPI・HTTP受信＋トークン認証）※Step3
  prompts/prompt_template.txt
  config/settings.toml     # 非機密設定（モデル名・親ドキュメントID・しきい値等）
  state/                   # ジョブ台帳（SQLite等）・クォータ台帳
  inbox/  work/  out/  logs/
```

### 依存・常駐運用（Nixで自己完結）
- 依存は `flake.nix`＋`flake.lock` で固定。**グローバル `pip` 等でのインストールは禁止**（devenvシェル or `nix run` 経由でのみ実行）。
- 実行入口：バッチは `nix run .#draft -- <input>`、常駐は `nix run .#server`。
- 常駐プロセス（FastAPIのHTTP受口）は **user-level systemd（`systemctl --user`）** で起動・自動再起動。system全体の変更は行わない（443終端のCaddyのみ§13 Step3でシステム側に追加）。
- ログは `~/projects/minutes/logs/` に出力（秘密情報はマスク。12章）。失敗時はジョブ状態=failed＋リトライ方針（9章）。
- 秘密情報（AI Engineトークン・Backlog APIキー）は**環境変数**で渡す（`SAKURA_AI_TOKEN` / `SAKURA_AI_BASE_URL` / `BACKLOG_API_KEY`）。ファイル・リポジトリ・ログに置かない（12章）。
- 実装言語は Python（OpenAI互換呼び出し・VTT解析が書きやすく、Nixで閉じ込めやすい）。PowerShell資産は移植。

## 6. 入口（トリガ）設計 ― 確定

前提（確定）：**この仕組みは他のメンバーも使える必要がある**。かつ**利用者は全員エンジニア**。
→ SSHは管理者IP限定（`110.4.190.167/32`・2201）なので、**A（SSH/CLI）は配布する入口にはできない**（他メンバーが使えない）。全員エンジニアのため**Web UIは不要**。

**決定：**
- **本番入口＝B2（PC側ツール `run.ps1` から VPSのHTTPS(443)へ送信して発火）**。利用者は今のPC操作の延長で1コマンド。
- **A（手動CLI/SSH）＝ 管理者の開発・検証用のみ**（配布しない）。
- **D（Backlog Webhook）＝ 入口ではなく出口の通知**（8章）。
- 認証は **トークン＋TLS（社内・在宅どちらでも有効）**。**IP制限は任意の追加ハードニング**（全員がオフィス固定IPからの利用と確定した場合に付与）。

| 方式 | 概要 | 位置づけ |
|---|---|---|
| A. 手動CLI（SSH） | VPSにSSHして `nix run .#draft -- input` | 管理者の開発/検証用のみ |
| B. PC側ツールから送信（HTTP） | PCの `run.ps1` がVPSのAPI(443)へPOSTして発火 | ★本番入口（B2で確定） |
| C. VPSのフォルダ監視 | 受信フォルダに置くと発火 | 不採用（アップロード手段とトリガが分離し、常駐監視・途中ファイル検知・権限管理が増える） |
| D. Backlog Webhook受信 | Backlog操作で発火 | 入口ではなく出口通知 |

導入順：**§13 Step2（管理者が方式Aで処理一式を検証・完全フォルダ内）→ Step3（B2のHTTP受口を公開して全メンバーへ）**。

### 入口B2のセキュリティ要件（#3対応）
- 公開：`443 → Nginx/Caddy → localhost の内部アプリ`（内部ポートは外部公開しない。AI-OCR手順書の推奨構成に合わせる）。**TLS必須**。
- 認証：**利用者ごとに発行するトークン**（または `client_id + token`）をヘッダに付与。サーバー側は**平文でなくハッシュで保存**し、**失効・再発行・退職者の無効化**を運用要件にする（共有トークンは利用者識別ができず監査と両立しないため不可）。改ざん対策にHMAC署名 or 短命アップロードトークンも可。
- ネットワーク（任意）：全員がオフィス固定IPからの利用と確定した場合は **接続元IP許可リスト**を追加（在宅/社外があるなら付けず、トークン＋TLSで対応）。
- 入力検証：サイズ上限、拡張子/MIME検証（.vtt/音声/テキストのみ許可）、ファイル名サニタイズ。
- 濫用対策：レート制限、監査ログ（**発行トークンで利用者識別**・いつ・何を送ったか。秘密情報はマスク）。
- 実行制御（非同期ジョブ）：**同時実行数の上限（1〜N）＋待ちキュー**（キューはあくまで“同時実行の待ち”。ジョブ状態は§9）。**同一入力の二重投入は既存jobを返す**（冪等・§9）。**クォータ不足時は受付拒否＋利用者へ通知**（月跨ぎの保留はしない。§10。※月5回運用では通常発生しない）。
- 段階：初期は A（管理者SSH）で処理を固め、B2のHTTP受口は上記要件を満たしてから公開。

### なぜ「Caddy＋443例外」で、より制約寄りの代替にしないか
制約の強い案件なので、代替を採らない理由を残す（合意形成のため）。
- **SSHトンネル**：利用者にSSHを配れない前提（管理者IP限定）に反するため不可。
- **アプリに直接443待受権限（CAP_NET_BIND_SERVICE）**：TLS証明書の取得/更新・権限境界・監査の観点でCaddyに劣る（アプリがTLSと特権を抱える）。
- **高ポートを直接公開**：ファイアウォールが80/443のみ、かつ利用者URL・TLS運用が悪化するため不可。
→ 「入口インフラ(Caddy)は共有・システム管理／アプリ依存はフォルダ内」の分離が最も制約の意図に沿う。

### B2の実装スタック（確定）
- **リバースプロキシ／TLS：Caddy**（`443` 待受・**Let's Encryptで証明書自動取得/更新**・設定が簡潔）。
  - ★制約との関係（正直な整理）：`443`は特権ポートで待受にはシステム側（`configuration.nix`）の設定が要る＝**「グローバル依存を足さない」制約に対する唯一の例外**。これは議事録アプリ固有のツール依存ではなく、AI-OCRとも共用する「Webの入口インフラ」（AI-OCR手順書も `外部→443→Nginx/Caddy→localhost` を想定）だが、システムを触る以上、**勝手に入れず管理者の明示OKを得てから追加する**（§13 Step3のHTTP公開時にFastAPI受口と一緒に導入）。
  - アプリ本体・ツール依存（python/ffmpeg/fastapi 等）は `~/projects/minutes/` 内でNix自己完結（グローバル無変更）。**§13 Step2（方式AのCLI検証）はこの例外を使わず完全にフォルダ内で完結できる。**
- **受け口アプリ：FastAPI（Python）**を `localhost` の高ポート（例 `8000`）で待受。Caddyから転送。**トークン認証はアプリ層で検証**（監査ログも同層）。
- **処理起動：非同期ジョブ方式**。受信したら即 `job_id` を返し、裏で「文字起こし→草案→out/」を実行。クライアントは `job_id` で状態/結果をポーリング（§9のジョブ状態と一致）。長時間処理のHTTPタイムアウトを回避。
- **証明書：** **推奨はドメイン取得＋Let's Encrypt（Caddy自動TLS）**。ドメインが無い場合の自己署名は、各PCで信頼設定が要り**配布運用の負債**になるため暫定扱い。実装着手は妨げないが、**§13 Step3（全メンバー公開）の前提条件**として「ドメイン取得 or 社内CA/自己署名の配布方式の決定」を置く。ACMEチャレンジ方式（HTTP-01／TLS-ALPN-01／DNS-01のいずれか）も公開前に確定する。
- **拡張余地：** B1（Webアップロード画面）は同じFastAPI/Caddyに画面を足すだけ、将来のGraph API自動取り込みは別プロジェクト。

> 上長メモの「Backlog Webhookの発火」は**入口ではなく出口側の通知**として整理（8章）。入口の本命はB2。

## 8. 出口（Backlog登録）と公開前チェックの整合 ★重要（#1・#2対応）

未チェックの草案をBacklogの共有領域へ自動登録すると、公開前チェックの前提が崩れる。したがって出口は次のとおり分離する。

- **既定（自動範囲）**：VPSは草案を `out/<meeting_id>.md` に出力するだけ（状態=draft_ready）。**Backlogには自動登録しない。**
- **本登録（出口＝Backlog Document API）**：人が公開前チェックを終え、承認メタデータ（例：`state/<meeting_id>.approval.json`。チェック実施者・日時・可否）が存在する場合のみ、`register.py` がBacklogへ登録する。登録前に同名・同ハッシュ確認（9章）を行い、重複を防ぐ。
- **通知（≠出口）**：Backlog Webhook は「Backlog上のイベントを外部へ知らせる」仕組み。VPSからBacklogへ送るものではない。用途は「本登録後にBacklog Webhookが関係者へ通知を発火」または「VPSからチャット等へ通知」。この2つを混同しない。

> どうしてもBacklog上でレビューしたい場合は、「非公開のレビュー専用領域に限定」または「本登録後のみ」に分ける。全社共有に近い親ドキュメント直下へ未チェック草案を置かない。

## 9. ジョブ・状態管理／冪等性・重複防止（#5対応）

複数会議・再送・タイムアウト後の再試行で、二重生成・二重登録が起きないようにする。

- **識別子**：`meeting_id`（録画名の `YYYYMMDD_HHMMSS` トークン由来）＋入力ファイルの SHA-256。同一入力の再投入は同一ジョブとして扱う（冪等）。
- **状態遷移**：`received → queued → running → draft_ready → approved → registered`（同時実行の空き待ちは `queued`／異常は `failed`／入力不正・クォータ不足での受付拒否は `rejected`）。ポーリングAPI・監査ログ・再投入時の返却はこの状態名で表す。台帳は `state/`（SQLite等）で管理。
- **排他**：`meeting_id` 単位のロック（ロックファイル or DBトランザクション）で同時二重処理を防止。
- **重複登録防止**：Backlog登録前に、既存の同名ドキュメント／同ハッシュ済み登録を確認（PC版の重複チェックを踏襲）。削除不可アカウントのため、誤登録は取り消せない前提で二重登録を厳に防ぐ。
- **チャンク単位の冪等性（#4対応）**：長音声は「chunk 1成功・chunk 2失敗」が普通に起きる。会議単位で丸ごと再実行すると成功済みチャンクを再送してクォータを浪費し、結合順序事故も起きる。そこで `meeting_id + audio_hash + chunk_index + chunk_hash` を保存し、各チャンクに状態 `pending/running/succeeded/failed`・API呼び出し回数・取得済みtextを持たせる。**再実行は未成功チャンクのみ**送る。結合は `chunk_index` 昇順で、**全チャンク succeeded のときだけ**実行する。
- **リトライ**：外部API失敗は指数バックオフで限定回数リトライ（429/5xx対応）。文字起こしリトライはクォータ消費に反映（10章）。

## 10. 無料枠（クォータ）の運用管理（#4対応）

さくらAI Engineの無料枠、特に**音声文字起こし50回/月**の消費を把握する。

前提（確定）：勉強会は**月最大5回**。1会議が30分超だと事前分割で複数回リクエストになる（例：45分→約2回）。それでも月5会議×約2回＝**約10回/月**で、50回に対し十分な余裕がある。そのため**厳格な事前ゲート（上限で自動停止）は必須ではなく、簡易な記録と警告で足りる**。

- **消費記録**：`state/quota.*` に月次・会議IDごとの呼び出し回数（Whisper／Chat別）を記録。「1会議＝1回」とは限らない（再実行・長時間分割）前提でカウント。
- **実行前の見積り（#6対応）**：処理開始前に **予定チャンク数＋最大リトライ分** を見積もり、当月残枠が不足する場合は**開始前に警告／停止を選べる**ようにする（90分会議・30MB超過による追加分割・リトライで消費が増える最悪ケースに備える）。
- **月次リセット基準**：カウントの**リセット基準日・タイムゾーン**（例：毎月1日 00:00 JST。実際はさくら側の基準に合わせる）を明記し、月替わりの数え間違いを防ぐ。
- **警告（保険）**：残回数がしきい値（例：残り10回）を切ったら警告。**上限到達時は新規受付を拒否＋利用者へ通知**（キューでの月跨ぎ保留はしない。§6と一致。通常は発生しない想定）。
- **可視化**：月次の利用台帳を出力（残枠の見える化）。
- 将来、会議数や1会議あたりの分割回数が増えて枠に近づくなら、その時点で有料枠を検討する。

## 11. LLM入力のトークン上限対策（#6対応）

長時間会議では、Teams .vtt ＋ Whisper の2ソース併記がモデルのコンテキスト上限を超え得る。

- **見積り**：投入前に入力トークン量を概算し、上限を超える場合は分割処理に切り替え。
- **分割・階層要約**：議題／時間帯単位で中間要約 → 最終統合。切り捨て（truncation）で内容を落とさない。
- **切り捨て禁止チェック**：入力が上限超過なのに単純切り詰めで送る事故を防ぐガード。
- **呼び出し回数**：長時間会議は Chat を複数回呼ぶ前提で見積る（3,000回/月に対しては十分だが、台帳に計上）。

## 12. セキュリティ・ガバナンス（#3・#8対応）

- 使用AIは承認済みのみ：さくらAI Engine（小見山さん承認）。外部AI/外部サーバー利用は上長承認が前提（本件は承認済み）。
- データ取扱い：さくらAI Engineは**国内クラウド完結・学習利用なし**。音声・文字起こし・草案は**承認済みAPI（さくらAI Engine）へは送信される**（社内VPS内で完結はしない）。**国外／未承認AIへは送信しない**。「外部送信なし」とは表現しない。
- 社内情報（録画・議事録）は外部同期される場所に置かない。VPSは社内管理下のNixOS。
- 公開前チェックは人手を維持（品質・コンプライアンスの最終担保）。未チェック草案はBacklog共有領域へ出さない（8章）。
- VPS作業は `~/projects/minutes/` 配下で自己完結。アプリのツール依存はグローバルに足さない（Nix devenv/flake、`flake.lock`固定、グローバルpip禁止）。
  - **唯一の例外**：HTTPS/443終端のCaddy等の「入口インフラ」はシステム側設定が要るため、**管理者の明示OK後に §13 Step3（HTTP公開）で追加**する（アプリ依存とは分けて扱う。6章参照）。§13 Step2（CLI検証）は完全にフォルダ内で完結。

### 秘密情報・権限（強化）
- 実行は**専用OSユーザー**（最小権限）で行う。他プロジェクトと権限を分離。
- Backlog APIキーは**最小権限**（登録に必要な範囲）。削除不可の運用キーと管理用キーを分離。
- 秘密の種類で管理を分ける：
  - **システム秘密（`SAKURA_AI_TOKEN` / `BACKLOG_API_KEY`）** … VPS側の**環境変数のみ**。**注入手順・ローテーション手順**を定める（誰がどこで設定するか）。
  - **利用者トークン（B2認証・6章）** … 平文ではなく**台帳にハッシュ保存**（環境変数ではない）。発行・失効・退職者無効化を運用。
- ログ・プロセス環境からの漏えい対策：ログの秘密情報**マスク**。エラー出力にトークンを含めない。
- `SAKURA_AI_BASE_URL` は**許可リスト検証**（想定ホスト以外への送信を防ぐ）。
- 監査ログ：入口の受付・登録・失敗を記録（秘密情報はマスク）。
