# 議事録自動化 フロー整理（2026-07-14）

## 0. この文書の位置づけ

上長助言「**認証認可より先に、アプリ全体のフローを整理する**」に沿って、議事録自動化の
**全体フロー**と、それに従属する**トリガ（起点）・認証認可**を整理する。
サーバー化設計書 v0.7 の **§6 入口 / §7 取り込み** を、今日判明した事実で見直す位置づけ。

**今日判明した重要な事実（設計前提の更新）**
- **録画(mp4)** は、Teams会議時に **OneDrive/SharePoint へ自動保存**される（Microsoft既定＋実運用で確認）。
- **字幕(vtt)** は、mp4と**同じフォルダには存在しない**（実機で確認済み）。Teamsの**トランスクリプト**として
  会議に紐づき、現状は **Teamsから手動ダウンロード**、Phase 2 では **Graph のトランスクリプトAPI**
  （`onlineMeetings/transcripts` 等）で取得する想定。→ mp4と字幕は**取得元が別**。
- その後、録画した人が**手動でCybozuにアップロード**している（全社閲覧用）。
- v0.7 は「録画はCybozu・**Cybozu APIなし**＝自動取得は将来課題」としていたが、
  **録画の原本はSharePoint側にもあり、SharePoint/OneDriveはGraph APIで取得可能**。
  → 「自動取得できない」という前提は（少なくとも録画については）誤り。**OneDrive/SharePoint（Teams録画の保存先）を起点にすれば自動化の道がある。**
  ※保存先は会議種別で異なる：**通常会議＝主催者のOneDrive／チャネル会議＝チームのSharePoint**（実機で確定が必要）。
- 実装は v0.7 の B2(HTTP/443) ではなく、**手動SSH＋CLI（`run.sh`）で稼働中**（第2回登録済み）。

---

## 1. 現状フロー（実装済み・稼働中）

```
1. Teams会議
2. →【自動】録画mp4 が OneDrive/SharePoint に保存（字幕vttは同フォルダに無い）
3. →【手動】録画者が Cybozu にアップロード（全社閲覧用の保管＋議事録の録画リンク元）
4. →【手動】録画(mp4)を入手（Cybozu/OneDrive）＋字幕(vtt)を Teams のトランスクリプトからDL
5. →【手動】scp で VPS の inbox/ へ
6. →【手動起動】./run.sh（音声抽出→さくらWhisper→用語訂正→gpt-oss-120bで下書き→整形）
7. →【人】out/minutes.md を公開前チェック
8. →【手動】register（Backlog Document APIで登録）
```

- **3〜5 は「同じファイルの移し替え」**で重複手作業。原本は最初からSharePointにある。
- 処理(6)・チェック(7)・登録(8)は完成・実運用済み（設計書 §3〜§4 のとおり）。

---

## 2. 現状の課題

- **重複手作業**：SharePoint→Cybozu→DL→scp と、同じ録画を何度も移動している。
- **アクセス配布が重い**：他メンバーが手動運用するには、VPSへのSSH（**公開鍵＋固定グローバルIP許可**）が要る。固定IPが無い人は運用に乗れない。
- **起動が手動**：会議のたびに人が scp→run.sh を実行。

---

## 3. トリガ（起点）方式 × 認証認可 ― フローが認証を決める

| トリガ方式 | 認証・認可の主体 | 録画の入手 | MS依存 | 実装規模 |
|---|---|---|---|---|
| **手動SSH＋CLI（現状）** | **SSH鍵＋許可IP**（自前） | 手動DL(Cybozu) | なし | 実装済み |
| 直接HTTPエンドポイント(443)＝旧B2 | **自前の発行トークン＋TLS**（＋任意IP） | PCからPOST | なし | 中〜大（認証設計） |
| Teamsチャット/コマンド起点(Graph/Bot) | **M365アカウント**（Teams認証がそのまま） | Graph API | 強 | 大・要権限/承認 |
| **SharePoint/OneDrive配置→hook（本命候補）** | **M365アプリ登録**（VPSアプリの資格情報） | **Graph APIで自動取得** | 強 | 大・要権限/承認 |

**要点**：Microsoft系（Teams/SharePoint）に乗せると、**利用者の操作はTeamsで録画するだけ**になり、
**利用者ごとのSSH鍵/IP配布は不要**。ただし**VPS側はEntra IDアプリ（app-only）としてGraphにアクセス**するため、
実際のアクセス範囲は「会議参加者かどうか」だけでは決まらず、**M365管理者が付与するアプリ権限・対象サイト/ユーザー・
Teamsアプリケーションアクセスポリシーで制御**する。逆に手動SSH/直接HTTPを続けるなら自前の認証（SSH+IP or トークン）が要る。

## 4. 目標フロー（段階計画）

### Phase 1（現在）：手動 SSH＋CLI ― つなぎ運用
- いま稼働中。当面はこれで回す。
- 他メンバーに配るなら、**公開鍵＋固定IPの許可**が必要（固定IPが無い人は「運用者に録画を渡す」でカバー）。

### Phase 2（本命）：OneDrive/SharePoint の Teams録画保存先を Graph API で監視・取得
```
1. Teams会議
2. →【自動】録画mp4 が保存（通常会議＝主催者OneDrive／チャネル会議＝SharePoint）。字幕は Teams トランスクリプトとして会議に紐づく
3. →【自動 or 定期ポーリング】VPSが新規録画を検知（Graph API）
4. →【自動】VPSが **mp4（Drive API）＋字幕（会議のトランスクリプトAPI）** を取得 → 処理（既存のsrc/一式）
5. →【人】公開前チェック（Backlogの下書き or 通知）
6. →【手動】register（公開前チェックのため人手は残す：設計書 §8）
```
- まず**自社の会議種別ごとの保存先（主催者OneDrive／SharePoint）を実機で確定**する（通常会議は主催者OneDriveに散る点に注意）。
- **scp・DL が消え、起動も自動化**。
- **利用者側の認証（SSH/IP・自前トークン）は不要**になり、アクセス制御は**M365管理者のアプリ権限設定に集約**される。
- インバウンドを晒したくなければ **Webhookでなく定期ポーリング**（VPSからの外向き通信のみ）で可。月5回ならポーリングで十分。
- ※旧B2（自前トークンHTTP）は、Graphが使えるなら不要になりうる。
- **要検証（Phase 2着手前）**：mp4と字幕は**取得元が別**（OneDrive/SharePoint の Drive と `onlineMeetings/transcripts`）。
  **会議と録画ファイルの対応付け**（どの録画がどの会議か）、**トランスクリプトの取得可否・必要権限**を実機で確認する。

## 5. 認証・認可（フロー確定後に決まる）

| | Phase 1（手動SSH） | Phase 2（SharePoint/Graph） |
|---|---|---|
| 認証の主体 | SSH鍵＋固定IP許可 | M365アプリ登録（VPSの資格情報） |
| 利用者の操作 | 各自SSHで実行 | 何もしない（録画するだけ） |
| 誰が設定 | 管理者（configuration.nix） | **M365管理者**（アプリ登録・権限付与） |
| 承認 | 許可メンバー/IPの合意 | **管理者作業＋上長承認**（外部連携・組織データアクセス） |
| アクセス制御 | 鍵とIPを持つ人 | 管理者が付与したアプリ権限・対象サイト/ユーザー・Teamsアクセスポリシー |

## 6. Cybozuの扱い（録画リンク）

議事録に貼る「録画リンク」は、全社が閲覧できる **Cybozu** のURLを使っている（設計書 §3）。
SharePointの録画は権限的に全社閲覧できないことが多い。

- **(あ) Cybozuは残す**：全社閲覧リンクはCybozu、**処理の入力だけSharePointから自動取得**。
  → 手動は「Cybozuへ1回アップロード」だけに減る。
- **(い) SharePointに寄せる**：全社が閲覧できるなら録画リンクもSharePointにしてCybozu廃止。
  → 全社閲覧の可否確認が必要。

## 8. Phase 2 技術検証・実装項目（詳細）

> Graph API の仕様・権限は変わりうるため、着手前に**最新のMicrosoft公式ドキュメントと実機**で確認する（本節は現時点の一般知識に基づく計画）。

### 8.1 認証（Entra ID アプリ登録）
- **Entra ID（Azure AD）にアプリを登録**し、**アプリケーション権限（app-only／無人実行）**で使う。
- 資格情報は**クライアントシークレット or 証明書**（証明書推奨）。VPSに保管（環境変数/鍵管理）。
- **管理者による「管理者の同意（admin consent）」が必須**。＝一般ユーザーだけでは完結しない。

### 8.2 必要な権限（最小化を狙う）
| 用途 | 権限（application） | 備考 |
|---|---|---|
| 録画mp4の取得 | `Files.Read.All` または **`Sites.Selected`** | `Sites.Selected` なら**特定サイトのみ**に絞れる＝最小権限・承認が通りやすい |
| 会議トランスクリプト | `OnlineMeetingTranscript.Read.All` | **さらに Teams の「アプリケーションアクセスポリシー」**（`Set-CsApplicationAccessPolicy`）で対象ユーザーを限定する運用が必要になることが多い |
| 会議情報（対応付け用） | `OnlineMeetings.Read.All` 等 | 会議ID解決に要る場合 |

→ ネックは **`.All` 系の広い権限**。**チャネル会議など録画が特定SharePointサイトに集まるなら `Sites.Selected` で絞れる可能性**があり承認が通りやすい。ただし**通常会議が主催者のOneDriveに散る場合は `Sites.Selected` だけでは足りず**、対象OneDriveへの権限設計（広めの `Files.Read.All` 等）を別途検証する。

### 8.3 取得の2系統と「対応付け」課題（最大の検証点）
- **録画(mp4)**：Drive（組織者のOneDrive or チームのSharePointサイト）の `Recordings` フォルダから。**どのDrive/サイトか特定**が要る。
- **字幕(vtt)**：`/users/{組織者}/onlineMeetings/{meetingId}/transcripts/{id}/content`（vtt形式）で取得。
- **課題**：**録画ファイル ↔ 会議(meetingId) ↔ トランスクリプト の対応付け**。**まず Graph の録画/トランスクリプトの metadata にある `meetingId` / `contentCorrelationId` で対応付けできるか確認**し、使えない場合に**ファイル名・開催時刻・組織者でフォールバック**する。**ここがPhase 2最大の検証ポイント**。

### 8.4 トリガ（起点）
- **ポーリング（推奨）**：Drive の **delta クエリ**で新規録画を定期検知。**インバウンド公開不要**（VPSからの外向き通信のみ）。月5回ならこれで十分。
- Webhook（Graph変更通知）：即時だが **443の受け口＋検証応答＋購読の更新**が要る。今回は不要。※OneDrive for Business の driveItem 通知は購読対象に制限があるため、採用時は要検証。

### 8.5 実機で確認すべきこと（チェックリスト）
- □ Teams録画の**実際の保存先**（組織者のOneDrive か チームのSharePoint か）
- □ **トランスクリプトがGraphで取得できるか**（会議の種類・保持設定次第で存在しないことがある）
- □ `OnlineMeetingTranscript.Read.All` ＋ **アプリケーションアクセスポリシー**の要否
- □ **録画↔会議の対応付け**が機械的に可能か（メタデータ／命名規則）
- □ 録画フォルダだけに権限を絞る **`Sites.Selected`** が使えるか
- □ **アプリ登録・権限付与・管理者同意ができる管理者は誰か**

### 8.6 リスク・折衷案
- 会議トランスクリプトAPIは**権限・前提が厳しめ**で、取得できないケースがある。依存要因の例：
  - Teams管理センター側で **Graph transcript access / 話者属性（speaker attribution）** を無効化していると **403 や属性欠落**になる。
  - **会議の有効期限切れ・ライブイベント非対応**など、会議種別・保持設定に依存する。
  → その場合は **「録画の取得＋起動の自動化だけ先行し、字幕は当面手動DLのまま（または話者属性なし形式）」** という折衷が可能（段階導入）。
- つまりPhase 2は「**録画自動化（比較的容易）**」と「**字幕自動化（要件が重い）**」に分けて考えられる。
