# minutes（VPS版）運用手順

録画と字幕から議事録の下書きを作り、人がチェックしてから Backlog へ登録するまでの
手順です。処理はすべて **VPS（AI-OCR用サーバー・NixOS）** の `~/projects/minutes/` で
動きます。コマンド先頭の **[PC]**＝Windowsの PowerShell、**[VPS]**＝SSHで入ったサーバー側。

- 送信先AI：さくらAI Engine（承認済み・国内完結・学習利用なし）
- 登録先：Backlog `tcs-s.backlog.com` の「議事録」フォルダ
- 前提：全員が同じ `aiocr` アカウントでVPSに入る運用（`~/.config/minutes.env` は共有・初回1回だけ設定）。
  ※各自別ユーザーの場合は、各自のホームに 0-1 のファイルを作る。

---

## ★ クイックリファレンス（毎回の作業・2回目以降）

初回セットアップ（下の A）が済んでいれば、毎回はこれだけ。

```bash
# 1) [PC] 録画と字幕をVPSへ（サイボウズからDL後。別のPowerShellで）
scp "<録画>.mp4" tcs-ocr:~/projects/minutes/inbox/rec_20260703_165826.mp4
scp "<字幕>.vtt" tcs-ocr:~/projects/minutes/inbox/teams.vtt

# 2) [VPS] 入って作業環境へ（トークン入力は不要）
ssh tcs-ocr
cd ~/projects/minutes && nix develop

# 3) [VPS] 下書き生成（会議番号は自動採番）
./run.sh                 # inbox/ の録画+字幕を自動で拾って draft（推奨・打ち間違い防止）
#   ↑ 追加指定したい時: ./run.sh --date 2026/07/03  /  ./run.sh --no 5
#   ↓ 手打ちしたい場合は従来どおり:
# python -m src.main draft inbox/rec_20260703_165826.mp4 --vtt inbox/teams.vtt

#    ↓ out/minutes.md を人がチェック・修正（タイトルは先頭の「# 見出し」）

# 4) [VPS] 登録：まず確認（送らない）→ 問題なければ本登録（y で送信）
python -m src.main register --dry-run
python -m src.main register
```

- 録画ファイル名に日時 `YYYYMMDD_HHMMSS` が入っていれば開催日を自動判定。無ければ `--date 2026/07/03` を付ける。
- 会議番号を手で決めたいときは `--no 5`。省略時はBacklogの「第N回」最大+1で自動採番。
- `nix develop` を抜けても `~/.config/minutes.env` のトークンは有効（毎回入れ直し不要）。

困ったら → 末尾の「トラブルシュート」。以降は詳細です。

## A. 初回セットアップ（新しく使う人・一度だけ）

### A-0. [PC] SSH接続の準備
PCの `~/.ssh/config` に `Host tcs-ocr`（HostName／User aiocr／Port 2201／IdentityFile）が
設定済みであること。`ssh tcs-ocr` でVPSに入れれば準備OK。

### A-1. トークン2本を用意
- **さくらAI Engine トークン**：`<UUID>:<シークレット>` の1本（コロン**有り**・77文字前後）。
- **Backlog APIキー**：Backlog → 個人設定 → API で発行（コロン**無し**の英数字1本）。
  ※さくらトークンとは別物。ここを取り違えると登録時に `HTTP 401` になる。

### A-2. [VPS] トークンを環境ファイルに保存（共有 aiocr なら全員ぶん1回）
```bash
ssh tcs-ocr
mkdir -p ~/.config
umask 077                                   # 以降作るファイルを本人だけ読める権限に
cat > ~/.config/minutes.env <<'EOF'
SAKURA_AI_TOKEN=＜UUID:シークレット＞
BACKLOG_API_KEY=＜Backlogのコロン無しキー＞
EOF
chmod 600 ~/.config/minutes.env             # 本人のみ読み書き（念のため明示）
```
- `＜...＞` を実値に置換してから貼る。`config.py` が起動時に自動読込する。
- このファイルは**ホーム内だけ**。リポジトリや `scp` で配らない（秘密情報）。
- 別パスにしたい場合は実行時に `MINUTES_ENV=/path/to/file` を指定。
- 確認は権限と長さだけ（中身は表示しない）：
  ```bash
  ls -l ~/.config/minutes.env                       # -rw------- なら適切
  ```

### A-3. [VPS] 読み込めるか確認（値は表示しません）
```bash
cd ~/projects/minutes && nix develop
python -c "from src import config; print('SAKURA set:', bool(config.SAKURA_TOKEN), '/ BACKLOG set:', bool(config.BACKLOG_API_KEY))"
```
→ `SAKURA set: True / BACKLOG set: True` なら完了。以降トークン入力は不要。

## B. サーバーへの配置・更新（管理者/初回構築時・普段は不要）

すでにVPSに `~/projects/minutes/` がある場合、この節は飛ばしてよい（普段の運用には不要）。
雛形を新規に載せる／更新する時だけ実施する。

### B-1. [PC] 転送先の確認
```powershell
ssh tcs-ocr "mkdir -p ~/projects"
ssh tcs-ocr "test -e ~/projects/minutes && echo EXISTS || echo OK"
```
→ OK なら未作成。EXISTS なら中身を確認（上書きは慎重に）。

### B-2. [PC] 雛形フォルダを転送
```powershell
scp -r "C:\Users\r-nakamura\source\議事録自動化\vps\minutes" tcs-ocr:~/projects/
```
※ ソースの一部ファイル（例 `src/config.py`）だけ更新するなら、そのファイルを個別に `scp`。

### B-3. [VPS] 権限を絞って環境を起動
```bash
ssh tcs-ocr
cd ~/projects/minutes
chmod 700 ~/projects/minutes     # 社内情報を他ユーザーから隠す
nix flake lock                   # 依存バージョンを固定（flake.lock作成）
nix develop                      # 初回はPython/ffmpeg取得で数分（止まって見えても正常）
```

### B-4. [VPS] 道具の確認（nix develop 内で）
```bash
python --version
ffmpeg -version | head -1
python -c "import requests; print('requests ok')"
```
期待：`Python 3.x` ／ `ffmpeg …` ／ `requests ok`

## C. 各コマンドの詳細

### C-1. draft（文字起こし＋下書き生成）
```bash
python -m src.main draft <録画> [--vtt <字幕.vtt>] [--no N] [--date YYYY/MM/DD]
```
- 処理：音声抽出 → 30分超なら無音区間で分割 → さくらWhisper文字起こし →
  Teams字幕(話者)とWhisper(文言)の2ソースでプロンプト作成 → さくらAI（gpt-oss-120b）で下書き。
- 出力：`out/minutes.md`。確認は `head -20 out/minutes.md`。
- `--no` 省略時は Backlog から自動採番（親配下「第N回」の最大+1／同一日付が既にあればその番号を再利用／取得不可なら「要確認」）。
- 必ず `~/projects/minutes` の中で実行する（`src/` とテンプレートを参照するため）。
- **`./run.sh`** … 上記 draft の便利ラッパー。`inbox/` の録画+字幕を自動で拾って draft する（追加オプションはそのまま渡せる：`./run.sh --date 2026/07/03`）。録画/字幕が複数あるとエラーで止めて取り違えを防ぐ。register は含めない（人チェック後に別途）。
  - **draft成功後、使った録画/字幕は `processed/<録画名>/` へ自動退避**され、`inbox/` は次の会議用に空になる（毎回の手動削除は不要）。削除ではなく移動なので後から参照可。失敗時は退避せず inbox に残す（再試行用）。

### C-2. register（人チェック後の Backlog 登録）
`out/minutes.md` を**人が確認・修正**してから実行。タイトルは本文先頭の `# 見出し`。
```bash
python -m src.main register [--dry-run] [--md <path>] [--title "..."] [--force] [--allow-unchecked]
```
- `--dry-run` … 送らずにタイトル・宛先・チェック結果だけ表示（まずこれ）。
- 何もオプションが無ければ確認プロンプト（`y` で送信）。
- `--md` 別ファイル指定 ／ `--title` タイトル上書き ／ `--force` 確認省略（重複/番号チェックは維持）。
- `--allow-unchecked` … 安全弁の解除（非常時のみ）。重複チェックが**走らない**時に登録を続けるほか、`[NUMBER CONFLICT]`（番号衝突）が**検出された場合も突破**して登録する。使用は慎重に。

チェック結果の意味：
- `[DUPLICATE]` … 同名文書が既存 → 登録中止（重複を防止）。
- `[NUMBER CONFLICT]` … 同じ「第N回」を別文書が使用 → 番号を直す。
- `[ERROR] duplicate check could not run` … Backlogへ到達不可 → **既定で中止**（安全側）。

登録後は Backlog を開いて目視確認。**このアカウントは追加はできるが削除はできない**ので、
`--dry-run` での事前確認を習慣に。誤登録時は管理者へ連絡。

## D. メンバーの追加・離任（共有 aiocr アカウント運用）

全員が同じ `aiocr` アカウントでVPSに入る運用のため、アクセスは各自の **SSH公開鍵**で管理する。

### D-1. メンバーを追加する（管理者・1回）
新メンバーのSSH公開鍵を `aiocr` の `authorized_keys` に追記する（誰の鍵か分かるよう行末に氏名コメント）。
```bash
# [VPS] aiocr で。本人にPCで ssh-keygen してもらい、公開鍵(~/.ssh/id_ed25519.pub)の中身を貼る
echo 'ssh-ed25519 AAAA...（本人の公開鍵） 山田太郎' >> ~/.ssh/authorized_keys
```
本人はPCの `~/.ssh/config` に `Host tcs-ocr`（A-0）を設定すれば `ssh tcs-ocr` で入れる。
トークン・フォルダは共有済みなので、以降はクイックリファレンス通りに使える。

### D-2. メンバーが離任する（退職・異動時・必須）
共有アカウントのため、離任者は**共有トークンを知り得た前提**で、鍵の削除に加えてトークンをローテーションする。
1. その人のSSH公開鍵を `authorized_keys` から削除（アクセス遮断）。
   ```bash
   nano ~/.ssh/authorized_keys      # 該当氏名コメントの行を削除して保存
   ```
2. **さくらAI Engine トークンを再発行**し、旧トークンを失効。`~/.config/minutes.env` の `SAKURA_AI_TOKEN` を新値に更新。
3. **Backlog APIキーを再発行**し、旧キーを失効。`~/.config/minutes.env` の `BACKLOG_API_KEY` を新値に更新。
4. 更新後、A-3 の確認（`SAKURA set: True / BACKLOG set: True`）を実行し、`./run.sh` が通ることを確認。

## トラブルシュート
| 症状 | 対処 |
|---|---|
| `SAKURA set: False` / `BACKLOG set: False` | `~/.config/minutes.env` のキー名（`SAKURA_AI_TOKEN`/`BACKLOG_API_KEY`）と値を確認。`ls -l` で存在も確認 |
| 登録で `HTTP 401 Authentication failure` | `BACKLOG_API_KEY` がさくらトークン（コロン有り77文字）になっていないか。Backlogの**コロン無し**キーに直す |
| `[WARN] ... group/other-accessible` | `chmod 600 ~/.config/minutes.env` |
| `nix develop` が experimental features エラー | `nix --extra-experimental-features 'nix-command flakes' develop` |
| 初回DLが遅い/失敗 | VPSから cache.nixos.org / github.com への到達を確認 |
| テンプレート未検出 / 変な work・out ができる | `~/projects/minutes` の中で実行しているか確認 |
| 開催日が「要確認」になる | 録画名に `YYYYMMDD_HHMMSS` が無い → `--date 2026/07/03` を付ける |
| `./run.sh` が Permission denied | scpで実行権限が落ちた → `chmod +x run.sh`（または `bash run.sh` で実行）|
| `別の処理が実行中です（draft/register）` | 同時実行を防ぐロック。共有環境のため**1会議ずつ**。前の処理の終了を待って再実行 |

## 注意（セキュリティ）
- 音声・文字起こしはさくらAI Engine（承認済み・国内完結・学習利用なし）へ送信される。
- `inbox/ work/ out/` の中身は社内情報。外部同期される場所に置かない。
- トークンは `~/.config/minutes.env`（600・ホーム内）だけに置く。チャット・メール・リポジトリに貼らない。
- 権限が緩い（group/other可）場合、読み込みは**警告を出しつつ続行**する（強制停止はしない）。`[WARN] ... group/other-accessible` が出たら必ず `chmod 600 ~/.config/minutes.env` で直す。所有者が自分でない/symlink/通常ファイルでない時は安全のため読み込まない。
- 未チェックの下書きを共有領域に出さない。Backlog登録＝公開、が唯一の出口。
- **共有 `aiocr` 運用の割り切り**：トークンは同じ `aiocr` を使う全員で共有される（同一UIDのため相互に読める）。`chmod 600` は他OSユーザー対策であり、チームメンバー間の分離ではない。離任時は必ず D-2 でローテーションする。
- **同時実行しない**：`inbox`・`out/minutes.md` は共有のため、2人が同時に処理すると取り違えが起きる。ロックで同時起動は弾くが、**1会議ずつ順番に**行う（特に他の人がレビュー中は `./run.sh` を実行しない）。登録者を個人名義にしたい場合は `BACKLOG_API_KEY=<自分のキー> python -m src.main register` で各自キーを使える。
