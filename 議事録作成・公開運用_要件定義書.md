# Teams録画を基にした議事録作成・公開運用 要件定義書

## 1. 目的

本要件定義書の目的は、Teams録画を基に議事録を作成し、全社員が後から会議内容を確認できる状態を整備することである。

単発で第1回システム設計×AI駆動開発勉強会の議事録を作成するだけでなく、今後の会議でも継続して利用できる議事録作成運用を確立する。

また、将来的な対応ではなく、初回から録画取得・文字起こし・AIによる議事録案作成・Markdown整形・Backlog登録までのエンドツーエンド自動化を前提とし、この一連の流れを標準化する。公開前確認のみ人間が行う。

---

## 2. 背景

第1回システム設計×AI駆動開発勉強会について、Teams録画を基に議事録として整理し、Backlogの全社共通（ALL）のドキュメント配下へ登録したいという依頼がある。

また、今後も同様の会議が発生する可能性があるため、都度手作業で対応するのではなく、再利用可能な手順および自動化可能な仕組みとして整備する必要がある。

---

## 3. 対象範囲

### 3.1 対象とする作業

本要件定義で対象とする作業は以下のとおりとする。

1. Teams録画を取得する
2. 録画の音声をテキスト化する
3. 文字起こし内容を基にAIで議事録案を作成する
4. 議事録をMarkdown形式に整える
5. 全社員向けに公開して問題ないか確認する
6. Backlog全社共通（ALL）のドキュメント配下へ登録する
7. 上記作業を手順化し、手順書を作成する
8. 公開前確認を除き、録画取得からBacklog登録までをエンドツーエンドで自動化する

---

## 4. 対象外

以下は今回の対象外とする。

* 動画を目視で確認し、発言内容の正確性を検証する作業
* 会議内容そのものの正誤判断
* 発言者ごとの厳密な発言録作成
* 議事録公開後のBacklog運用ルール全体の見直し
* TeamsやBacklogの権限設計全体の変更

---

## 5. 前提条件

本運用は以下を前提とする。

* 元データはTeams録画またはTeams録画から取得できる音声データとする
* 議事録の正確性確認は、動画を見直して行わない
* 文字起こし結果を議事録作成の主な根拠とする
* AIで作成された議事録案は、そのまま公開せず、人間が公開可否を確認する
* 出力形式はMarkdown形式とする
* 最終的な登録先はBacklogの全社共通（ALL）のドキュメント配下とする
* 全社員が閲覧する前提のため、機密情報・個人情報・不適切表現が含まれていないかは確認する

## 7. 業務フロー

### 7.1 初回対応フロー

第1回システム設計×AI駆動開発勉強会については、以下の流れで対応する。

1. Teams録画を取得する
2. 録画から音声データを取得する
3. 音声を文字起こしする
4. 文字起こし結果をAIに入力する
5. AIで議事録案を作成する
6. Markdown形式に整える
7. 全社員向けに公開して問題ないか確認する
8. Backlog全社共通（ALL）のドキュメント配下へ登録する
9. 実施した作業を手順書に反映する

## 8. 自動化フロー

本運用では、録画取得からBacklog登録までをエンドツーエンドで自動化する。以下を標準フローとして実装する。

1. Teams録画が保存される
2. 録画ファイルまたは音声ファイルを取得する（自動取得できない場合は手動アップロード）
3. 音声を自動で文字起こしする
4. 文字起こし結果をAIに渡す
5. AIが議事録案を生成する
6. Markdown形式に自動整形する
7. 公開前確認用の議事録案を作成する
8. 担当者が内容を確認し、承認結果を `approval.json` に記録する
9. 承認済みであれば、`approval.json` を検証したうえでBacklog全社共通（ALL）のドキュメント配下へ自動登録する

公開前確認のみ人間が行い、それ以外の工程は自動実行する。
承認が完了した場合に限りBacklog登録を自動実行し、承認前に自動登録することはない。
理由は、全社員向けに公開する資料であり、機密情報や個人情報が含まれる可能性があるためである。

## 9. 機能要件

### 9.1 Teams録画取得

Teamsで録画された会議データを取得できること。

取得対象は以下とする。

* 録画ファイル
* 音声データ

なお、文字起こしはTeams側のトランスクリプトを使用せず、録画から抽出した音声を基に行う（詳細は `音声テキスト化方式選定.md` を参照）。

### 9.2 文字起こし

録画または音声データから、議事録作成に利用できる文字起こしデータを作成できること。

文字起こしでは以下を取得する。

* 会議全体の発言内容
* 話題の流れ
* 決定事項
* 依頼事項
* 今後の対応事項

なお、発言者名の完全な特定は必須としない。

### 9.3 AIによる議事録案作成

文字起こし内容を基に、AIで議事録案を作成できること。

AIに作成させる内容は以下とする。

* 会議名
* 開催日時
* 会議の目的
* 議題
* 内容要約
* 決定事項
* TODO・担当・期限
* 補足事項
* 次回以降の確認事項

なお、参加者は全社員向け公開の観点から標準フォーマットに欄を設けない。個人名の記載可否は `Markdownフォーマット定義.md`「6.11 機微情報の扱い」に従う。

### 9.4 Markdown整形

議事録はMarkdown形式で出力できること。

Markdownの標準フォーマット、各セクションの記載ルール、ファイル名・タイトル規則は `Markdownフォーマット定義.md` を正とし、本書では二重に定義しない。

標準の章構成は以下とする（詳細は `Markdownフォーマット定義.md` を参照）。

```md
# YYYYMMDD_会議名_議事録

## 1. 会議概要

## 2. 会議の目的

## 3. 議題

## 4. 内容要約

## 5. 決定事項

## 6. TODO・対応事項

## 7. 確認事項・要確認事項

## 8. 補足事項

## 9. 備考
```

### 9.5 公開前確認

Backlogへ登録する前に、全社員向けに公開して問題ない内容か確認できること。

確認観点は、`check_items`（詳細は「13. 公開前チェックリスト」）と対応させ、以下とする。

* 個人情報が含まれていないか（`personal_information`）
* 機密情報が含まれていないか（`confidential_information`）
* 社外秘・部門限定・未公開などの限定公開情報が含まれていないか（`internal_only_information`）
* 未確定事項を決定事項として書いていないか（`uncertain_decisions`）
* TODOの担当者・期限に誤りがないか（`todo_owner_due_date`）
* 全社員向けに公開して問題ない表現になっているか（`public_safe_expression`）

不要な雑談や内部事情が含まれていないかは、`public_safe_expression` の確認に含める。

### 9.6 Backlog登録

確認済みのMarkdown議事録を、Backlogの全社共通（ALL）のドキュメント配下へ登録できること。

登録時には以下を整理する。

* 登録先プロジェクト
* 登録先ドキュメント階層
* タイトル命名ルール
* 登録日
* 会議名
* 関連資料の有無

## 10. 非機能要件

### 10.1 正確性

動画を目視で確認して、文字起こしや議事録内容を細かく照合する作業は行わない。

ただし、AIが作成した議事録について、公開に問題がないかは人間が確認する。

### 10.2 再利用性

第1回システム設計×AI駆動開発勉強会だけでなく、今後の会議でも同じ流れで議事録を作成できること。

そのため、手順書には具体的な操作手順、使用するプロンプト、確認観点を記載する。

### 10.3 保守性

自動化処理を行う場合でも、一部処理が失敗した際に手動で対応できるようにする。

例：

* 録画取得に失敗した場合は手動アップロードで対応
* 文字起こしに失敗した場合は別ツールで再実行
* AI出力が不十分な場合はプロンプトを修正して再生成
* Backlog登録に失敗した場合はMarkdownを手動登録

### 10.4 セキュリティ

録画、音声、文字起こし、議事録案には社内情報が含まれる可能性があるため、取り扱いに注意する。

特に以下を考慮する。

* 社外ツールへアップロードしてよい情報か確認する
* AIツールに入力してよい内容か確認する
* 一時ファイルの保存場所を明確にする
* 不要になった録画・音声・文字起こしファイルの扱いを決める

本書ではフォーマットの詳細を二重に定義せず、`Markdownフォーマット定義.md` を正とする。

標準の章構成は以下とする（詳細は `Markdownフォーマット定義.md` を参照）。

1. 会議概要
2. 会議の目的
3. 議題
4. 内容要約
5. 決定事項
6. TODO・対応事項
7. 確認事項・要確認事項

9. 備考

## 12. AIに入力するプロンプト要件

AIには、文字起こし内容とあわせて以下の条件を指定する。

```text
あなたは社内会議の議事録作成担当です。

以下のTeams会議の文字起こし内容を基に、全社員向けに公開する議事録を作成してください。

条件：
- 出力形式はMarkdown
- Backlogのドキュメントに貼り付ける前提
- 全社員が後から会議内容を確認できるようにする
- 発言録ではなく、要点を整理した議事録にする
- 雑談や不要な発言は省略する
- 決定事項、TODO、確認事項を分ける
- 未確定事項を決定事項として書かない
- 個人情報・機密情報に見える内容は「要確認」として扱う
- 動画を見直した正確性確認は行っていない前提で記載する
- 不明な箇所は推測せず「要確認」と書く

出力構成（詳細は Markdownフォーマット定義.md に従う）：
# YYYYMMDD_会議名_議事録
## 1. 会議概要
## 2. 会議の目的
## 3. 議題
## 4. 内容要約
## 5. 決定事項
## 6. TODO・対応事項
## 7. 確認事項・要確認事項
## 8. 補足事項
## 9. 備考

文字起こし内容：
---
ここに文字起こし内容を貼り付ける
---
```

## 13. 公開前チェックリスト

公開前確認の観点は `Markdownフォーマット定義.md`「6.9 公開前確認の扱い」に従い、機械判定は `Backlog通信定義.md` の `approval.json`（`check_items`）で行う。

`check_items` と確認観点の対応は以下とする。

| check_items | 確認観点 |
|---|---|
| `personal_information` | 個人情報が含まれていない |
| `confidential_information` | 機密情報が含まれていない |
| `internal_only_information` | 社外秘・部門限定・未公開などの限定公開情報が含まれていない |
| `uncertain_decisions` | 未確定事項を決定事項として記載していない |
| `todo_owner_due_date` | TODOの担当者・期限に誤りがない |
| `public_safe_expression` | 全社員向けに公開して問題ない表現になっている |

登録処理では、`check_items` の全項目が `true` である場合のみ登録を許可する。

登録前には、以下も確認する。

- [ ] Markdown形式になっている
- [ ] Backlogに貼り付けても崩れない
- [ ] 登録先が全社共通（ALL）のドキュメント配下である
- [ ] タイトルが命名ルールに沿っている

## 15. 自動化方針

初回から自動化を前提とし、録画取得からBacklog登録までをエンドツーエンドで自動化する。公開前確認のみ人間が行い、それ以外の工程は自動実行する。

### 15.1 自動化対象

| 処理         | 自動化方針                                   |
| ---------- | --------------------------------------- |
| Teams録画取得  | 自動取得する。自動取得できない場合は手動アップロードで代替する         |
| 音声抽出       | 自動化する                                   |
| 文字起こし      | 自動化する                                   |
| AI議事録生成    | 自動化する                                   |
| Markdown整形 | 自動化する                                   |
| 公開前確認      | 人間が確認する（`approval.json` に承認結果を記録する）     |
| Backlog登録  | 承認後に自動登録する（`approval.json` の検証を通過した場合のみ） |

### 15.2 手動確認を残す箇所

以下は完全自動化せず、人間による確認を残す。

* 全社員公開してよいかの確認
* 機密情報・個人情報の確認
* 決定事項として扱ってよいかの確認
* TODOの担当者・期限の確認

上記はすべて公開前確認として実施し、その結果を `approval.json` に記録する。この承認が完了して初めてBacklog登録が自動実行される。Backlog登録処理自体に、承認とは別の手動ゲートは設けない。

## 16. 運用上の役割

| 役割         | 担当内容                           |
| ---------- | ------------------------------ |
| 議事録作成担当    | 録画取得、文字起こし、AI議事録案作成、Markdown整形 |
| 内容確認者      | 全社員向け公開可否の確認                   |
| Backlog登録処理管理者 | 自動登録処理の監視、失敗時の例外対応、APIキー管理      |
| 運用管理者      | 手順書更新、自動化改善、ルール整備              |

なお、本書の「内容確認者」、`Markdownフォーマット定義.md` の「確認者」、`Backlog通信定義.md` の「承認者」は、同一の役割を指す。`approval.json` の `approved_by` には、その承認者1名のIDを記録する（単数承認）。承認者リストは、この `approved_by` を照合するための資格者一覧であり、役割そのものではない。

## 17. 受け入れ条件

本対応は以下を満たした時点で完了とする。

* 第1回システム設計×AI駆動開発勉強会の議事録がMarkdown形式で作成されている
* 議事録が全社員向けに公開して問題ない内容として確認されている
* Backlog全社共通（ALL）のドキュメント配下へ登録できる状態になっている
* 録画から議事録生成までの手順書が作成されている
* 今後の会議でも同じ手順を再利用できる
* 録画取得からBacklog登録までをエンドツーエンドで自動実行できる

* いずれかの工程が失敗した場合に、手動で対応できる構成になっている
* 手動確認が必要な箇所が明確になっている

## 19. 結論

本件では、第1回システム設計×AI駆動開発勉強会の議事録を作成するだけでなく、今後も使える議事録作成運用を整備することを目的とする。

Teams録画を起点として、文字起こし、AIによる議事録案作成、Markdown整形、公開前確認、Backlog登録までを一連の標準プロセスとして定義する。

正確性については、動画を目視で確認して検証するのではなく、文字起こし内容を基に議事録を作成する前提とする。

ただし、全社員向けに公開する資料であるため、Backlog登録前には必ず人間が公開可否を確認する。

初回から自動化を前提に設計しつつ、公開前確認などリスクの高い工程は手動確認を残すことで、効率化と安全性を両立する。
