## Markdown形式のフォーマット定義

### 1. 基本方針

本件で作成する議事録は、Backlogのドキュメントに登録することを前提として、Markdown形式で作成する。

Markdownは、全社員が後から会議内容を確認しやすいように、発言録ではなく要点整理型の議事録として記載する。

また、AIで生成した議事録案をそのまま登録するのではなく、人間による公開前確認を行ったうえで、確認済みMarkdownをBacklogへ登録する。

本フォーマットでは、以下を重視する。

* 会議の目的が分かること
* 議題と内容要約が整理されていること
* 決定事項が明確であること
* TODO・対応事項が整理されていること
* 不明点や確認が必要な事項を推測せず明示すること
* 全社員向けに公開して問題ない表現であること
* Backlog上で表示が崩れにくいこと

---

### 2. Markdown作成ルール

議事録Markdownは、以下のルールで作成する。

| 項目      | ルール                   |
| ------- | --------------------- |
| 文字コード   | UTF-8                 |
| BOM     | なし                    |
| 改行コード   | LF                    |
| 見出し     | `#`、`##`、`###` を使用する  |
| 表       | Markdownテーブルを使用する     |
| 箇条書き    | `-` を使用する             |
| 番号付きリスト | `1.`、`2.` を使用する       |
| 強調      | 必要最小限にする              |
| HTMLタグ  | 原則使用しない               |
| 画像埋め込み  | 原則使用しない               |
| 外部リンク   | 必要な場合のみ記載する           |
| コードブロック | コマンドや設定値を記載する場合のみ使用する |

Backlog登録時に本文差分やハッシュ確認を行いやすくするため、Markdown本文はUTF-8、BOMなし、LF改行に正規化する。

Backlogへ登録する本文は、承認時にSHA-256を算出した正規化済みMarkdown本文と同一のものを使用する。

---

### 3. ファイル名ルール

議事録Markdownファイル名は、以下の形式とする。

```text
YYYYMMDD_第〇回システム設計×AI駆動開発勉強会_議事録.md
```

例：

```text
20260706_第1回システム設計×AI駆動開発勉強会_議事録.md
```

議事録Markdownファイル名は、MarkdownのH1（議事録タイトル）およびBacklog登録タイトルと一致させる。
会議概要の「会議名」は、議事録タイトルから日付と `_議事録` を除いた名称と対応させる（詳細は「4. タイトル・会議名の定義」を参照）。

---

### 4. タイトル・会議名の定義

本運用では、以下を区別して扱う。

| 項目            | 意味                       | 例                        |
| ------------- | ------------------------ | ------------------------ |
| 会議名           | 会議そのものの名称                | 第1回システム設計×AI駆動開発勉強会                 |
| 議事録タイトル       | 日付・会議名・種別を含む議事録名         | 20260706_第1回システム設計×AI駆動開発勉強会_議事録    |
| Markdownファイル名 | 議事録タイトルに `.md` を付けたファイル名 | 20260706_第1回システム設計×AI駆動開発勉強会_議事録.md |
| Backlog登録タイトル | Backlogドキュメントのタイトル       | 20260706_第1回システム設計×AI駆動開発勉強会_議事録    |

MarkdownのH1見出しには、議事録タイトルを記載する。

```md
# 20260706_第1回システム設計×AI駆動開発勉強会_議事録
```

会議概要の「会議名」には、日付や `_議事録` を含めない純粋な会議名を記載する。

```md
| 会議名 | 第1回システム設計×AI駆動開発勉強会 |
```

ファイル名、MarkdownのH1、Backlog登録タイトルは一致させる。
会議概要の「会議名」は、議事録タイトルから日付と `_議事録` を除いた名称と対応させる。

議事録タイトル・ファイル名・Backlog登録タイトルは、`Backlog通信定義.md`「9.2 タイトル正規化」のルール（NFC、前後空白削除、連続空白統一など）で正規化したうえで一致させる。

### 5. 標準フォーマット

議事録Markdownは、以下の構成を標準とする。

```md

## 1. 会議概要

| 項目 | 内容 |
|---|---|
| 会議名 |  |
| 開催日 | YYYY/MM/DD |
| 開催時間 | HH:mm〜HH:mm |
| 開催方法 | Teams |
| 登壇者 |  |
| 作成日 | YYYY/MM/DD |
| 作成者 |  |
| 確認者 |  |
| 登録先 | Backlog 全社共通（ALL）ドキュメント配下 |

## 2. 会議の目的

- 

## 3. 議題

1. 

## 4. 内容要約

### 4.1 

- 

## 5. 決定事項

| No | 決定事項 | 補足 |
|---|---|---|
| - | 該当なし | - |

## 6. TODO・対応事項

| No | 内容 | 担当 | 期限 | 状態 |
|---|---|---|---|---|
| - | 該当なし | - | - | - |

## 7. 確認事項・要確認事項

| No | 確認内容 | 理由 | 確認先 |
|---|---|---|---|
| - | 該当なし | - | - |

文字起こし結果には誤認識が含まれる可能性があるため、人名、会社名、社内用語、日付、数字、担当者、期限、決定事項、TODOは確認対象とする。
```

### 6. 各セクションの記載ルール

#### 6.1 会議概要

会議概要には、議事録の基本情報を記載する。

| 項目      | 記載内容                        |
| ------- | --------------------------- |
| 会議名     | Teams会議名、または議事録用に整理した会議名    |
| 開催日     | 会議の開催日                      |
| 開催時間    | 分かる場合のみ記載する                 |
| 開催方法    | 原則 `Teams`                  |
| 登壇者     | その会で説明・登壇した人（スピーカー）。全参加者一覧は記載しない。不明な場合は `要確認` |
| 作成日     | 議事録Markdownを作成した日           |
| 作成者     | 議事録作成担当者                    |
| 確認者     | 公開前確認を行った担当者                |
| 登録先     | `Backlog 全社共通（ALL）ドキュメント配下` |

開催時間、確認者などが不明な場合は、推測せず `要確認` と記載する。
ただし、確認者欄の `要確認` はドラフト作成時のみ許容し、Backlog登録時点では確認者（承認者）を必ず記載する。

なお、全参加者の一覧は標準フォーマットに設けず、その会の登壇者（スピーカー）のみを「登壇者」欄に記載する。登壇者は公開の場での説明者にあたるため記載してよいが、個人名の扱いは「6.11 機微情報の扱い」に従い、異動・評価など機微な文脈の個人名は本文に断定して書かない。

本フォーマットの「確認者」は、`Backlog通信定義.md` の承認者（`approved_by` ／承認者リスト）と同一とする。

---

#### 6.2 会議の目的

会議の目的には、会議が何のために開催されたのかを簡潔に記載する。

記載例：

```md
## 2. 会議の目的

- 第1回システム設計×AI駆動開発勉強会として、今後の進め方、参加者への依頼事項、確認事項を共有すること。
```

目的が文字起こしから明確に判断できない場合は、以下のように記載する。

```md
## 2. 会議の目的

- 要確認：文字起こし内容から会議目的を明確に判断できないため、確認が必要。
```

---

#### 6.3 議題

議題には、会議内で扱われた主なテーマを番号付きで記載する。

記載例：

```md
## 3. 議題

1. 勉強会の目的説明
2. 今後の進め方
3. 参加者への依頼事項
4. 質疑応答
```

議題が明確に区切られていない場合は、文字起こし内容から話題単位で整理する。
ただし、推測で過度に細かく分割しない。

---

#### 6.4 内容要約

内容要約には、議題ごとに要点を整理して記載する。

発言録のように逐語的に書くのではなく、全社員が後から内容を把握できる粒度で要約する。

記載例：

```md
## 4. 内容要約

### 4.1 勉強会の目的説明

- 本勉強会では、今後の進め方と参加者に求める対応について説明された。
- 詳細な手順やスケジュールについては、別途案内される予定である。

### 4.2 今後の進め方

- 今後の対応は、段階的に進める方針である。
- 具体的な日程や担当は、必要に応じて確認する。
```

内容要約では、以下に注意する。

* 雑談は原則記載しない
* 発言者ごとの逐語録にしない
* 未確定事項を決定事項として書かない
* 不明な内容は `要確認` と記載する
* 社外秘・個人情報に該当しそうな内容は記載せず、必要に応じて `要確認` とする

---

#### 6.5 決定事項

決定事項には、会議内で確定した内容のみを記載する。

記載例：

```md
## 5. 決定事項

| No | 決定事項 | 補足 |
|---|---|---|
| 1 | 次回以降もTeams録画を基に議事録を作成する | 詳細手順は別途整理する |
```

決定事項として扱ってよいか不明な内容は、決定事項には記載せず、確認事項に記載する。

記載例：

```md
## 7. 確認事項・要確認事項

| No | 確認内容 | 理由 | 確認先 |
|---|---|---|---|
| 1 | ○○を正式な決定事項として扱ってよいか | 文字起こし上では確定か判断できないため | 要確認 |
```

---

#### 6.6 TODO・対応事項

TODO・対応事項には、会議後に対応が必要な作業を記載する。

記載例：

```md
## 6. TODO・対応事項

| No | 内容 | 担当 | 期限 | 状態 |
|---|---|---|---|---|
| 1 | Teams録画を取得する | 要確認 | 要確認 | 未着手 |
| 2 | 議事録案を作成する | 議事録作成担当 | YYYY/MM/DD | 未着手 |
```

担当者または期限が文字起こしから判断できない場合は、推測せず `要確認` と記載する。

状態は、原則として以下のいずれかを使用する。

| 状態  | 用途               |
| --- | ---------------- |
| 未着手 | これから対応する事項       |
| 対応中 | すでに対応中の事項        |
| 完了  | 会議後に完了済みと確認できた事項 |
| 要確認 | 状態が判断できない事項      |

---

#### 6.7 確認事項・要確認事項

確認事項には、文字起こし内容だけでは判断できない内容や、公開前に確認すべき内容を記載する。

記載例：

```md
## 7. 確認事項・要確認事項

| No | 確認内容 | 理由 | 確認先 |
|---|---|---|---|
| 1 | 登壇者として誰を記載すべきか | 文字起こしから話者を正確に判断できないため | 要確認 |
| 2 | ○○を決定事項として扱ってよいか | 発言上は検討段階の可能性があるため | 要確認 |
```

以下に該当する場合は、確認事項として扱う。

* 人名が不明確
* 担当者が不明確
* 期限が不明確
* 決定事項か検討事項か判断できない
* 数字、日付、金額が不明確
* 社内用語や略語の意味が不明確
* 全社員向けに公開してよいか判断できない内容がある

---

#### 6.8 補足事項

補足事項には、議事録本文には入れにくいが、後から確認する際に有用な情報を記載する。

記載例：

```md
## 8. 補足事項

- 本議事録はTeams録画の音声を文字起こしした内容を基に作成している。
- 発言者ごとの厳密な発言録は作成対象外である。
```

---

#### 6.9 公開前確認の扱い

本節は公開前確認の運用仕様であり、ここで扱う `approval.json` 等の管理情報は、生成される議事録本文（備考を含む）には含めない。

公開前確認は、Markdown本文内のチェックボックスでは管理しない。

理由は、Markdown本文内の `- [ ]` を `- [x]` に変更すると、Markdown本文のSHA-256が変わり、承認時本文とBacklog登録本文の一致確認に影響するためである。

そのため、公開前確認の実施結果は `approval.json` の `check_items` で管理する。

Markdown本文には、公開前確認のチェックリストを含めない。
公開前確認の管理方法（approval.json 等）は運用側の承認フローで扱い、議事録本文（備考を含む）には記載しない。

`approval.json` の `check_items` は以下とする。

```json
{
  "check_items": {
    "personal_information": true,
    "confidential_information": true,
    "internal_only_information": true,
    "uncertain_decisions": true,
    "todo_owner_due_date": true,
    "public_safe_expression": true
  }
}
```

各項目の意味は以下とする。

| check_items                 | 対応する確認観点               |
| --------------------------- | ---------------------- |
| `personal_information`      | 個人情報が含まれていない           |
| `confidential_information`  | 機密情報が含まれていない           |
| `internal_only_information` | 社外秘・部門限定・未公開などの限定公開情報が含まれていない |
| `uncertain_decisions`       | 未確定事項を決定事項として記載していない   |
| `todo_owner_due_date`       | TODOの担当者・期限に誤りがない      |
| `public_safe_expression`    | 全社員向けに公開して問題ない表現になっている |

`internal_only_information` は、全社員向けBacklogに載せてはいけない情報として、社外秘だけでなく以下も対象とする。

* 部門限定・チーム限定の情報
* 人事・評価・異動に関する情報
* 顧客契約・未公開案件に関する情報
* 金額・単価などの機微な数値
* 障害・インシデントの詳細

`public_safe_expression` は、以下のような表現を含まないことを確認する（NG例）。

* 差別的・攻撃的な表現
* 特定個人の評価や責任追及を含む表現
* 未確認情報を確定事項として断定する表現
* 部門外に文脈が伝わらない略語・内部用語

Backlog登録処理では、`check_items` の全項目が `true` である場合のみ登録を許可する。

公開前確認が未完了の場合、Backlog登録処理は実行しない。

---

#### 6.10 備考

備考には、議事録作成の前提や制約を記載する。

必ず以下の内容を含める。

```md
本議事録は、Teams録画から抽出した音声の文字起こし内容を基に作成している。  
動画を目視で確認した正確性検証は実施していない。
```

必要に応じて、以下も記載する。

```md
文字起こし結果には誤認識が含まれる可能性があるため、人名、会社名、社内用語、日付、数字、担当者、期限、決定事項、TODOは確認対象とする。
```

---

#### 6.11 機微情報の扱い

個人情報、機密情報、社外秘情報に該当する可能性がある内容は、本文に断定して記載しない。

また、確認事項・要確認事項にも、具体的な個人名、会社名、案件名、金額、識別情報などをそのまま記載しない。

確認事項に記載する場合は、内容を抽象化して記載する。

記載例：

```md
## 7. 確認事項・要確認事項

| No | 確認内容 | 理由 | 確認先 |
|---|---|---|---|
| 1 | 個人名を議事録に記載してよいか | 全社員向け公開範囲に含めてよいか判断が必要なため | 要確認 |
| 2 | 特定案件名を本文に記載してよいか | 公開範囲の確認が必要なため | 要確認 |
| 3 | 具体的な金額を記載してよいか | 機密情報に該当する可能性があるため | 要確認 |
```

記載しない例：

```md
| 1 | 山田太郎さんの異動予定を記載してよいか | 個人情報の可能性があるため | 要確認 |
| 2 | A社向け○○案件の金額1,000万円を記載してよいか | 機密情報の可能性があるため | 要確認 |
```

全社員向けに公開するMarkdown本文には、機微情報そのものを残さないことを原則とする。

---

#### 6.12 該当なしの場合の記載ルール

決定事項、TODO・対応事項、確認事項・要確認事項が存在しない場合は、空欄の行を残さず、`該当なし` と記載する。

##### 決定事項がない場合

```md
## 5. 決定事項

| No | 決定事項 | 補足 |
|---|---|---|
| - | 該当なし | - |
```

##### TODO・対応事項がない場合

```md
## 6. TODO・対応事項

| No | 内容 | 担当 | 期限 | 状態 |
|---|---|---|---|---|
| - | 該当なし | - | - | - |
```

##### 確認事項・要確認事項がない場合

```md
## 7. 確認事項・要確認事項

| No | 確認内容 | 理由 | 確認先 |
|---|---|---|---|
| - | 該当なし | - | - |
```

##### 補足事項がない場合

```md
## 8. 補足事項

- 該当なし
```

空の `No.1` 行は残さない。

---

#### 6.13 確認者の記載ルール

会議概要の「確認者」は、公開前確認を実施した承認者1名を記載する。

本運用では、1件の議事録に対する承認者は1名とする（単数承認）。

```md
| 確認者 | 山田 太郎 |
```

ただし、Backlog登録処理で承認者として判定する値は、Markdown本文内の確認者名ではなく、`approval.json` の `approved_by` と承認者リストのIDとする。

Markdown本文内の確認者名は表示用であり、機械判定には使用しない。

`approval.json` の `approved_by` は、以下の仕様とする。

| 項目      | 仕様                                    |
| ------- | ------------------------------------- |
| 形式      | 承認者リストの不変ID（文字列1つ）                     |
| 空値      | 空文字は不可                                |
| 照合      | IDが承認者リストに存在し、`active=true` であること     |
| 表示名との対応 | Markdown本文の確認者名と対応するが、判定には使用しない        |

---

### 7. 記載しない内容

全社員向けに公開する議事録であるため、以下は原則として記載しない。

* 不要な雑談
* 個人的な発言
* 個人情報
* 機密情報
* 社外秘情報
* 社内の未公開情報
* 発言者を特定する必要のない発言者名
* 確認できていない推測
* 文字起こし上あいまいな内容を断定した表現

記載すべきか判断できない場合は、本文に断定して書かず、確認事項に `要確認` として記載する。
その際も、具体的な機微情報（個人名・会社名・案件名・金額など）はそのまま書かず、「6.11 機微情報の扱い」に従い抽象化して記載する。

---

### 8. 表記ルール

Markdown議事録では、以下の表記ルールを使用する。

| 表記     | 用途                  |
| ------ | ------------------- |
| `要確認`  | 文字起こし内容だけでは判断できない内容 |
| `未定`   | 決まっていないことが明確な内容     |
| `該当なし` | 対象が存在しないことが明確な内容    |
| `未着手`  | TODOの状態             |
| `対応中`  | TODOが対応中の場合         |
| `完了`   | TODOが完了している場合       |

`要確認` と `未定` は使い分ける。

| 表記    | 意味                       |
| ----- | ------------------------ |
| `要確認` | 情報が不明、または文字起こしだけでは判断できない |
| `未定`  | 会議内でまだ決まっていないことが明確である    |

---

### 9. AI出力時の注意点

AIに議事録案を作成させる場合は、以下を守る。

* 不明な情報を推測で補完しない
* 決定事項と検討事項を混同しない
* TODOの担当者・期限を勝手に作らない
* 発言者名を必要以上に記載しない
* 雑談や不要な発言を削除する
* 全社員向けに公開できる表現へ整える
* 機密情報・個人情報に見える内容は本文に断定して書かず、確認事項に回す。その際も具体的な機微情報は書かず、「6.11 機微情報の扱い」に従い抽象化する

---

### 10. 受け入れ条件

Markdown形式のフォーマットは、以下を満たすこと。

* Markdown形式で作成されている
* UTF-8、BOMなし、LF改行で保存されている
* Markdown本文がSHA-256算出時と同じ正規化済み本文である
* Backlog登録タイトル、Markdownファイル名、MarkdownのH1が一致している
* 会議概要の会議名には、日付と `_議事録` を含めない純粋な会議名が記載されている
* 会議概要が記載されている
* 会議概要に登壇者（スピーカー）欄があり、全参加者一覧は記載していない（登壇者が不明な場合は `要確認`）
* 会議の目的が記載されている、または `要確認` と記載されている
* 議題が整理されている
* 内容要約が発言録ではなく要点整理になっている
* 決定事項が表形式で整理されている
* TODO・対応事項が表形式で整理されている
* 担当者や期限が不明な場合は `要確認` と記載されている
* 決定事項、TODO、確認事項がない場合は `該当なし` と記載されている
* 確認事項・要確認事項が明示されている
* 確認事項にも、個人情報・機密情報・社外秘情報そのものが記載されていない（記載する場合は抽象化されている）
* 登録時点で確認者が記載されている（`要確認` のままにしない）
* 確認者は1名が記載されている（単数承認）
* 承認者の機械判定は `approval.json` の `approved_by`（承認者リストのID・文字列1つ）で行う
* 公開前確認チェックリストをMarkdown本文内に含めていない
* 公開前確認結果は `approval.json` の `check_items` で管理されている
* `check_items` に `public_safe_expression` が含まれている
* 備考に「動画を目視で確認した正確性検証は実施していない」旨が記載されている
* 個人情報、機密情報、社外秘情報が含まれていないことを確認できる
* 承認時のMarkdown本文とBacklog登録本文が一致している
